# AGENTS.md — Onboarding & resume rules

> Read this file first. Then `progress.md`, then `plan.md`.
> This file is the **canonical resume-from-crash protocol** for this repo.

## 1. Resume-from-crash sequence (start of every session)

If the previous session was lost or you are a new agent / contributor,
**read in this exact order before doing any work**:

1. `AGENTS.md` (this file) — rules & protocol
2. `progress.md` — current phase, last session, next concrete action
3. `plan.md` — full project plan (phases, todos, decisions)
4. **Tail** of `prompts.md` (last 5–10 entries) — recent user intent in their own words
5. **Tail** of `decisions.md` (last 5–10 entries) — recent decisions with rationale

After this you should be able to resume any in-flight work without
asking the user "where were we?"

## 2. Per-session contract

### On every user prompt
- **Append verbatim** to `prompts.md` BEFORE doing the work.
  - Preserve original wording, language (EN / 中文 / mixed), line breaks.
  - Never summarize, paraphrase, translate, or "clean up".
  - For empty / image-only prompts, record `_[empty]_` and `[image: <filename>]`.
  - Group by `## YYYY-MM-DD — <short topic>` (newest section at the bottom).

### On every meaningful decision
- Append one bullet to `decisions.md`: `YYYY-MM-DD — <decision> — <one-line rationale>`.
- "Meaningful" = anything that constrains future work (tech stack, schema choices, scope changes, equivalence target, etc.).

### On every completed todo
- Tick the checkbox in `progress.md` "Phase status" section.
- `UPDATE todos SET status='done' WHERE id='<todo-id>'` in the SQL session DB.

### On every session end
- Update `progress.md` sections: **Last session** (date, #, summary) + **Next up** (concrete next action).
- A future agent should be able to act on "Next up" with zero ambiguity.

## 5. SQL session DB

Todos and dependencies live in the session DB (not in markdown).
Use the `sql` tool:

- Query ready todos:
  ```sql
  SELECT t.* FROM todos t
  WHERE t.status='pending'
  AND NOT EXISTS (
    SELECT 1 FROM todo_deps td JOIN todos dep ON td.depends_on=dep.id
    WHERE td.todo_id=t.id AND dep.status!='done'
  );
  ```
- Mark in-progress: `UPDATE todos SET status='in_progress' WHERE id='<id>';`
- Mark done: `UPDATE todos SET status='done' WHERE id='<id>';` AND tick in `progress.md`.

## 6.5 End-of-stage 自动测试 + 测试报告（强制）/ End-of-stage automated test + report (mandatory)

**触发条件 / Trigger**
- 每完成"一个阶段"工作就触发一次。这里"一个阶段"包括：
  - 阶段 1 中的任一章节小阶段：`stage1-toc`、`stage1-overall-flow`、`stage1-carrington`、`stage1-shellpoint`、`stage1-arvest`、`stage1-cc5`、`stage1-selene`、`stage1-mrc`、`stage1-sls`、`stage1-scattered`、`stage1-dataflow`、`stage1-review`。
  - 阶段 2 中的每个交付物：`stage2-feature-list`、`stage2-srs`、`stage2-pages`、`stage2-data-model`、`stage2-api`、`stage2-dev-plan`、`stage2-test-plan`、`stage2-phased-impl`。
- "Each stage" = every `stage1-*` chapter todo + every `stage2-*` deliverable todo above.

**强制顺序 / Mandatory order**
1. 完成本阶段产出文档 / 代码。
2. **运行测试套件**（见下方"测试矩阵"）。
3. **写测试报告**到 `test_reports/<todo-id>_YYYY-MM-DD.md`。
4. 报告中 P0 全部 PASS 才能把对应 todo `UPDATE … SET status='done'`。
5. 然后才能问用户"是否进下一章"。
- Do NOT mark the stage todo `done` until the report exists and all P0 checks pass.

**测试矩阵 / Test matrix**

| 阶段类型 / Stage kind | 必跑检查 / Required checks |
|---|---|
| 阶段 1 文档章节 / Stage-1 doc chapter | (a) `pytest`（回归保护，必须全绿） / pytest must stay green; (b) markdown 链接校验：本章引用的所有 `.md` 锚点 / 相对路径存在 / verify every internal markdown link exists; (c) 源码引用校验：本章每一条 `path/to/file.py:LINE` 真实可定位（grep 源仓库行号在范围内）/ every `path/to/file.py:LINE` citation resolves in the read-only source repo; (d) 中英双文档对齐：`.zh.md` 与 `.en.md` 章节标题层级一致 / `.zh.md` and `.en.md` share the same heading skeleton; (e) `mkdocs build --strict`（如果 mkdocs 还在用）/ `mkdocs build --strict` if mkdocs is still wired in. |
| 阶段 2 设计 / 代码 / Stage-2 design or code | (a) `pytest -q`；(b) `ruff check .`；(c) `mypy`（新代码 strict）；(d) 如有 schema / API 契约文件，跑契约 lint；(e) 文档链接 + mkdocs build。 |

**测试报告模板 / Report template** — 存到 `test_reports/<todo-id>_YYYY-MM-DD.md`，必须含：
1. Stage / todo-id / 日期 / 触发人 (agent vs user)
2. 范围（本阶段新增 / 修改的文件清单）
3. 每条检查的：命令、退出码、PASS/FAIL、关键输出节选
4. 失败项分级：P0（阻塞下一阶段）/ P1（本阶段内修）/ P2（记 backlog）
5. 结论：是否允许进入下一阶段
6. 下一阶段 todo-id + 已通知用户

**Sections in every report**: scope, per-check command + exit code + verdict, failure severity, gate decision, next stage id.

**与 SQL todo 状态联动 / Coupling with SQL todos**
- 报告未写 → 该 todo 必须停留 `in_progress`，不得 `done`。
- 报告写了但 P0 有 FAIL → 该 todo 改为 `blocked`，把失败摘要写进 description。
- 全 PASS → todo `done` + `progress.md` 勾选 + 通知用户。

## 6.7 文档头与图表说明（强制）/ Document header & diagram caption (mandatory)

**每一份业务文档（`docs/**/*.md`，含 `.zh.md` / `.en.md`）必须满足：**

1. **文档头** —— 在第一个正文标题之前，列出三项：
   - `> Purpose / 目的`：1-3 句说明本文档解决什么问题。
   - `> Intended audience / 目标读者`：列出主要读者（例如：validator 维护者 / 新成员 onboard / 阶段 2 系统设计师）。
   - `> Revision history / 版本历史`：一个表格，列 `Date | Author | Change`，倒序排列，首条总为当前最新更新。
2. **图表说明** —— 每张图（mermaid / 表格 / 流程图）必须紧跟一段 **caption**：
   - 一句话回答"这张图在讲什么"（不是重复标题）。
   - 标明来源（`Source: <file>:<line>`）。
3. **动作序列说明** —— 凡是图中包含步骤序列（sequenceDiagram、带方向箭头的 flowchart、有 `autonumber` 的图、或显式编号的列表步骤），必须在图后给出 **按编号的逐步解释**：
   - 序号与图中 `autonumber` 或人工编号一致。
   - 每步描述：触发条件 → 行为 → 输出/副作用 → 源码定位。
4. **测试**：`tools/stage_doc_checks.py` 在阶段结束测试中验证以上三项缺失会被标记 P0 失败，必须修复后才能进 done。
5. **图必须 inline**：当一份文档讲解某个工作流 / 数据流时，相关的图与 caption / step-by-step 解释必须**直接写在描述该流程的章节内**，**不允许**单独建一份"只放图"的 markdown 文件再让正文文档去引用。原则：图永远紧贴它所解释的那段文字。

**Every business-facing doc under `docs/**/*.md` (both `.zh.md` and
`.en.md`) MUST contain:**

1. **Doc header** — before the first H1 body section, three items:
   - `> Purpose`: 1-3 sentences on what problem this doc solves.
   - `> Intended audience`: who is expected to read this (e.g.,
     validator maintainers / onboarding engineers / Stage 2 system
     designers).
   - `> Revision history`: a `Date | Author | Change` table, newest
     first.
2. **Diagram caption** — every diagram (mermaid / table / flowchart)
   must be immediately followed by a caption paragraph:
   - One sentence answering "what is this diagram showing" (do not
     restate the heading).
   - A source citation `Source: <file>:<line>`.
3. **Action-sequence explanation** — whenever a diagram contains an
   action sequence (sequenceDiagram, directional flowchart, any
   `autonumber` diagram, or explicitly numbered steps), a **numbered
   step-by-step explanation** must follow:
   - Step numbers match the diagram's `autonumber` or manual numbering.
   - Each step describes: trigger → action → output / side effect →
     source location.
4. **Testing**: `tools/stage_doc_checks.py` will, at end-of-stage,
   verify that all three items are present. Missing any of them is a
   **P0 failure** and blocks marking the stage `done`.
5. **Diagrams stay inline**: when a doc explains a workflow or
   dataflow, the diagram + caption + step-by-step explanation **must
   live inside the same chapter that describes that flow**. Do **not**
   create a separate "diagrams only" markdown file and link to it from
   the main doc. A diagram belongs immediately next to the prose that
   explains it.

## 6.9 Mermaid node-ID legend rule (mandatory)

Any dataflow / pipeline mermaid diagram that uses short ID-style node names
(`T#`, `V#`, `SH#`, `R#`, etc.) — i.e. names that are **not** the real source
identifier — **must** be accompanied immediately after its figure caption by a
small "图例 / Legend" table that explains:

1. What each prefix stands for (one word: Table / Validator / Sheet / Rule …).
2. The numeric range used in this specific figure (e.g. `T1`–`T13`).
3. The mapping from each ID to its real source-code or business name
   (one bullet/cell per node, listed exhaustively for `V#` and `SH#`; `T#`
   may be summarized "appears in prose by real table name" if it would
   otherwise dominate the legend).

The legend must also state, in one sentence, that **node IDs are display-only
cross-references between the figure and the prose, not source-code
identifiers**. Inside each mermaid node, surface both the ID and the real
name using the `ID["ID · realname"]` convention from v1.1.

**Where this applies**: every per-servicer chapter (MRC, plus any future
servicer), the overall-flow chapter, and any new diagram added during Stage 2
design / SRS work.

**Rationale**: v9.1 user prompt (this session) — without the legend, readers
see `SH1` / `V2` in prose without any way to know what they mean or where to
find them in the figure or in the source code.

## 6.10 Diagram + text co-requirement (强制 / mandatory)

**Scope**: every workflow, dataflow, ETL process, validation flow, and
system-interaction description in this repo's documentation
(`docs/**/*.md`, plus any top-level explanatory `.md` that walks through
a flow).

**Rule**: such content MUST include **both** a visual diagram **and** a
written explanation. Neither substitutes for the other.

1. **Every important workflow / dataflow has at least one diagram.**
   Plain prose without a figure is not acceptable for flow-shaped
   content.
2. **Every diagram has a corresponding textual explanation covering**:
   - business purpose,
   - execution flow,
   - input / output,
   - key transformations,
   - important dependencies or assumptions.
3. **Diagram type may be any appropriate UML / system-analysis form**,
   including but not limited to: Flowchart, Sequence Diagram, Activity
   Diagram, State Diagram, Class Diagram, ERD, Data Lineage Diagram,
   Component Diagram. Pick the form that best fits the content.
4. **Complex logic uses two layers**:
   - first a high-level overview diagram of the whole flow;
   - then decomposed lower-level detail diagrams for each major
     sub-process. Each sub-diagram gets its own textual explanation per
     item 2.
5. **Priorities for both diagram and text**: readability, explainability,
   reverse-engineering transparency, onboarding friendliness, debugging
   efficiency.
6. **Reflect actual existing behavior**: diagrams and explanations must
   describe what the existing system actually does, not a hypothetical
   future design — unless the diagram/section is **explicitly marked**
   (e.g. heading prefix `[Proposed]` / `[Future]` or a clear caption
   note) as a future-state design.

**Interaction with § 6.7 and § 6.9**: this rule is additive. § 6.7's
caption + revision-history requirements and § 6.9's mermaid node-ID
legend rule still apply to every diagram introduced under § 6.10.

**Testing**: missing diagram for a flow-shaped section, or missing
textual explanation for a diagram, is a **P0 failure** in the
end-of-stage check and blocks marking the stage `done`.

## 6.11 Living Stage 1 docs + 3-tier behavior marker (mandatory)

User decision 2026-05-17 (G1 sign-off): MRC Stage 1 chapters
(`docs/mrc/1.0-toc → 1.6-baseline.{zh,en}.md`) are **provisionally
approved with iterative refinement allowed**. Stage 2 work proceeds in
parallel.

**Stage 1 docs are LIVING, not frozen.** When the user — or any Stage 2
investigation — discovers missing logic, incorrect assumptions,
undocumented edge cases, unclear business semantics, or validation
inconsistencies, the relevant Stage 1 chapter is updated **in place** and:

1. an entry is appended to the chapter's "Revision history" table (bump
   version, list the change);
2. a one-line entry is appended to `decisions.md` linking the change to
   its driver (user prompt / Stage 2 finding / external source);
3. a refinement test report is added under `test_reports/` mirroring the
   convention `<chapter-or-todo-id>_<YYYY-MM-DD>.md`;
4. the full test matrix is re-run (`tools/stage_doc_checks.py`,
   `pytest -q`, `mkdocs build --strict`) and results recorded.

**3-tier behavior marker** — every behavior assertion in the Stage 1
chapters must carry exactly one of these tier tags so readers can
distinguish epistemic status at a glance:

| Tier | Tag | Meaning |
|---|---|---|
| 1 — Verified | `[FROM-CODE]` (existing) or new `[CONFIRMED]` | Reverse-engineered from source code with line-range citation **and** corroborated against the physical baseline XLSX (V1–V12 sweep) |
| 2 — Inferred | `[FROM-CODE]` (without physical confirmation) or `[VERIFY]` | Reverse-engineered from source code only; no physical-artefact corroboration yet |
| 3 — Newly discovered | `[FOUND-DURING-STAGE2]` (new) | Surfaced after G1 sign-off while building Stage 2 (engine, renderer, harness, UI); must include date, finder, and reference to the Stage 2 todo or trace that surfaced it |

When a tier-2 `[VERIFY]` is upgraded to tier 1, update both the tag and
the chapter revision history. When a tier-3 `[FOUND-DURING-STAGE2]` is
added, also append to `decisions.md` so it does not silently drift back
into the "always-known" pile.

**Testing**: a Stage 1 chapter without per-assertion tier tags, or with a
contradiction between two tier-1 assertions, is a **P1 issue** flagged
during the end-of-stage check; it does not block Stage 2 progress but
must be reconciled in the next refinement pass.
