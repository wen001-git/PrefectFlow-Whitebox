# 10.0 验证策略（Stage 2）

---

## 1. 为何存在本文档

Stage 2 的理想验证方案（`plan.md §§ 4.2/4.4` 中的 G2a/G2b）需要：

1. 将所有 Redshift 输入**冻结为 Parquet 快照**（`baselines/mrc/2026-04-30/input_snapshots/`）。
2. 对该快照运行未修改的 legacy 代码，生成**冻结的基准 XLSX**。
3. 将新系统输出与冻结基准做**逐单元格对比**（cell-identity diff）。

这种方案具备可复现性、历史调试能力，且一旦快照存在就完全由 Agent 驱动。

**阻塞点**：Agent 和操作员目前均无法获取 Redshift 凭据、VPN 访问权限及 HashiCorp Vault 令牌，无法安排导出窗口。G2a（由操作员执行的导出步骤）处于**环境阻塞**状态。

为避免整个项目因等待凭据窗口而停滞，本文档固化了一种**临时实时对比策略**，该策略：

- 今日即可解锁 Stage 2 实施工作。
- 将长期 G2a/G2b/G3 理想方案作为明确的迁移目标予以保留。
- 定义新的硬关卡（**G2b-LIVE**），用于管控所有实施工作。

**工具现状**：所有 G2a 工具均已完成并等待使用（`tools/freeze_snapshot.py` v2.2、A1–A6 操作手册、`.env.example`）。一旦凭据窗口开放，按操作手册（`tools/docs/g2a-operator-runbook.{en,zh}.md`）执行即可关闭 G2a。

## 5. 对比运行操作员工作流

G2b-LIVE 关闭（C5 通过）后，标准对比工作流如下：

### 步骤 1 — 运行 legacy 系统

具有 Redshift VPN + Vault 凭据的操作员执行：

```bash
python tools/run_legacy_mrc.py --remit-date 2026-04-30
```

此脚本封装了 legacy `flow/remit_validation` MRC 入口点（参见操作手册 A5：
`tools/docs/g2a-operator-runbook.{en,zh}.md`）。输出：
`baselines/mrc/2026-04-30/legacy_run/validation_report.xlsx` 及运行元数据旁车 JSON。

### 步骤 2 — 运行新系统

操作员（或 Agent，实施后）执行：

```bash
python tools/run_newsystem_mrc.py --remit-date 2026-04-30
```

初期为 C3 存根，生成合成 XLSX。Stage 2 实施落地后调用真实引擎。
输出：`baselines/mrc/2026-04-30/new_run/validation_report.xlsx`。

### 步骤 3 — 执行对比

```bash
python tools/compare_validation.py compare \
  --legacy baselines/mrc/2026-04-30/legacy_run/validation_report.xlsx \
  --new    baselines/mrc/2026-04-30/new_run/validation_report.xlsx \
  --report baselines/mrc/2026-04-30/comparison_report.html
```

调用 C4 编排器（C4 完成后可用）。

### 步骤 4 — 阅读 HTML 报告并迭代

在浏览器中打开 `comparison_report.html`。报告将差异分类为：

- **值差异** — 单元格值或公式不同
- **样式差异** — 数字格式、字体、填充、边框、对齐差异
- **结构差异** — 缺行/多行、合并单元格变化、列宽 / 冻结窗格变化
- **表级差异** — 工作表顺序、缺表/多表、维度变化

在将实施 PR 标记为可评审之前，应解决主要值差异和结构差异。
每个涉及验证器的实施 PR 必须附带对比运行证据。

### 自动模式（快捷方式）

```bash
python tools/compare_validation.py compare \
  --remit-date 2026-04-30 --servicer mrc --auto \
  --report baselines/mrc/2026-04-30/comparison_report.html
```

`--auto` 自动按顺序执行步骤 1–3（需要 `.env` 中配置操作员凭据）。

## 6. 明确接受的缺陷

采用该临时策略，项目明确接受以下限制：

1. **跨天不可复现** — Redshift 源数据可能变化；即使代码不变，两次对比运行在不同日期可能产生合理差异。
2. **历史调试能力弱** — 若事后发现回归，在没有 G2a（冻结快照）的情况下无法重放精确的失败输入。
3. **操作员瓶颈** — 每次有意义的验证周期都需要操作员手动提供 Redshift 凭据；Agent 无法自主运行对比。
4. **共同缺陷盲区** — legacy 和新系统对真相都犯的错误（如共享上游缺陷）不会浮现；只有*两者之间的差异*才会被发现。
5. **跨天非确定性误报** — legacy 代码中的任何非确定性排序都会显示为误报差异；套件必须容忍或明确标记此类情况。

这些缺陷在此明确记录——而非掩盖——以便未来贡献者理解该策略下对比证据的认识论局限。
