## 1. What the gate is

The acceptance gate is the v9.1 mechanism that enforces the equivalence
bar from `plan.md` ("Equivalence bar (locked)"):

> **Cell-identical.** For a fixed `remit_date` baseline frozen from
> production, the new system's `validation_report.xlsx` matches the
> legacy XLSX byte-for-byte on sheet names, sheet order, column order,
> header rows, data row order, every cell value, and highlight colors /
> conditional formatting.

The gate has two surfaces:

| Surface | Use | Location |
|---|---|---|
| **Pytest suite** | Developer + CI signal during normal test runs | `tests/acceptance/mrc/` |
| **CLI tool** | One-shot CI invocation + operator-driven check | `tools/acceptance_gate.py` |

Both surfaces are kept deliberately equivalent — anything the pytest
suite checks, the CLI also checks (and vice versa). They share the
same engine wiring, the same `tools/xlsx_diff.py`, and the same
allowlist file.

## 2. Three operational modes

| Mode | When | What it diffs | Skip reason |
|---|---|---|---|
| **Self-consistency** | Always | Engine output ↔ Engine output (two runs) | Never skips — failure is a real bug |
| **Baseline diff** | When `baselines/mrc/<date>/validation_report.xlsx` exists | Engine output ↔ captured legacy XLSX | `ENV-SKIP` if the file is absent (G2a not yet closed) |
| **Legacy live diff** | When `ACCEPTANCE_LEGACY_LIVE=1` + Vault creds present | Engine output ↔ fresh legacy run in the same window | `ENV-SKIP` if creds absent |

Self-consistency is the **floor**: if two engine runs differ, every
other comparison is meaningless (the new system is a moving target).
The gate enforces this by escalating any self-consistency diff —
even a minor one — to the worst possible verdict.

## 3. Verdict vocabulary

`acceptance_verdict.json` always contains one of:

| Verdict | Exit code | Meaning |
|---|---|---|
| `PASS` | 0 | Every active component passed (skips don't count as failures) |
| `MINOR_DIFFS` | 1 | At least one non-floor component had only documented minor diffs |
| `MAJOR_DIFFS` | 2 | At least one component had a major diff, OR self-consistency violated |
| `ERROR` | 3 | Engine / runner / IO failure |
| `SKIPPED` | 4 | Self-consistency itself could not run (env-blocked) |

A `PASS` gate produces a json shaped like:

```json
{
  "verdict": "PASS",
  "exit_code": 0,
  "components": {
    "self_consistency": {"status": "PASS", "major": 0, "minor": 0},
    "baseline":         {"status": "SKIPPED", "reason": "--baseline not provided"},
    "legacy_live":      {"status": "SKIPPED", "reason": "--legacy-mode=skip"}
  }
}
```

This is exactly the shape returned by the smoke run in
`runs/acceptance-smoke/` (committed alongside the gate landing).

## 4. The self-consistency floor

`tests/acceptance/mrc/test_self_consistency.py` and
`tools/acceptance_gate.py` both run the engine twice and diff the two
XLSX. The contract:

* Both runs must produce the same `run_id` (deterministic id derived
  from `(servicer, remit_date, source.kind)`).
* Both XLSX files must produce a `0 major / 0 minor` diff via
  `tools/xlsx_diff.py`.

A floor violation means the engine or the renderer has hidden
non-determinism (timestamps, dict-iteration ordering, random tie-breaks).
Fix it before evaluating any other tier.

## 5. The allowlist

`tests/acceptance/mrc/acceptance_minor_diffs_allowlist.json` documents
the **only** acceptable minor diffs versus baseline / legacy. Schema
and policy live in
`tests/acceptance/mrc/allowlist_schema.md`. Key rules:

1. **MAJOR diffs are never allowlistable.** The dimensions `value`,
   `formula`, `merged_cells`, `structure` are rejected by the meta-test.
2. Every entry must carry an `ADR_ref` pointing to an entry in
   `decisions.md`.
3. The file starts empty (`[]`). The acceptance bar is strict
   cell-identity.

To add an entry:

1. Reproduce the diff (run the gate locally).
2. Write an ADR in `decisions.md` explaining why the new system
   intentionally diverges on that dimension.
3. Append the entry to the JSON array. Commit both changes in the same
   PR.

## 6. Operator runbook (local)

### 6.1 Smoke run (self-only)

```powershell
python tools\acceptance_gate.py `
  --servicer MRC `
  --remit-date 2026-04-30 `
  --legacy-mode skip `
  --output runs\acceptance-smoke\
```

Expected: `verdict=PASS`, exit code 0, baseline + legacy SKIPPED.

### 6.2 Baseline mode (after capturing a baseline)

See `baselines/mrc/2026-04-30/CAPTURE_INSTRUCTIONS.md` for the capture
procedure. Then:

```powershell
python tools\acceptance_gate.py `
  --servicer MRC `
  --remit-date 2026-04-30 `
  --baseline baselines\mrc\2026-04-30\validation_report.xlsx `
  --legacy-mode skip `
  --output runs\acceptance-with-baseline\
```

### 6.3 Full live mode (operator only)

```powershell
$env:ACCEPTANCE_LEGACY_LIVE = "1"
python tools\acceptance_gate.py `
  --servicer MRC `
  --remit-date 2026-04-30 `
  --baseline baselines\mrc\2026-04-30\validation_report.xlsx `
  --legacy-mode live `
  --output runs\acceptance-live\
```

Requires `.env` with `VAULT_ADDR` + `VAULT_TOKEN` per `.env.example`.

## 7. CI integration

`.github/workflows/acceptance-gate.yml` wires the gate into CI:

* **Every PR** runs the pytest tier + the CLI in self-consistency
  mode. Build fails on any floor violation.
* **`main` push and `workflow_dispatch`** also run the full gate
  (baseline diff when present; live-legacy when dispatched with the
  appropriate secrets).

### Required repository secrets (declared, not hardcoded)

| Secret | Purpose | Used by |
|---|---|---|
| `VAULT_ADDR` | HashiCorp Vault URL | live legacy step only |
| `VAULT_TOKEN` | Vault token with Redshift read perms | live legacy step only |

Self-consistency and structural-invariant tiers require **no secrets**
and run for every PR.
