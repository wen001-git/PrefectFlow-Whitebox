## 1. Design Principles

1. **Servicer discriminator on every shared model** — Every shared model (from
   `RawTableSnapshot` to `XlsxRenderJob`) carries a `servicer: ServicerId`
   field. Stage 2 **must not** hard-code the string `"MRC"` inside any model
   definition; servicer identity is injected via an enum value, not branched on
   internally. (See ch 1.1 § 5.1 — `flow == 'mrc'` dispatch scattered across
   5 locations; Stage 2 eliminates this scattering.)

2. **Immutable frames between pipeline stages (lineage preservation)** — Each
   pipeline stage (raw → normalized → intermediate → validated → payload →
   render-job) produces a **new immutable object** rather than mutating the
   previous stage's structure. Each object holds a back-reference to its
   immediate upstream, making every output cell's lineage traceable end-to-end
   back to raw table rows. (See ch 1.4 § 3.1 — 5-layer lineage diagram: raw →
   CTE → projection → stamp → render.)

3. **Explicit cell-coordinate carrying** — `SheetPayload` pre-computes every
   cell's `(row, col)` coordinate at construction time rather than deriving
   it at render time from DataFrame position. This allows `BaselineComparison`
   to perform cell-level diffs without invoking the renderer, and is a
   prerequisite for the UI drill-down feature. (See ch 1.6 § 3.1 — column order
   and tab order are fixed by source code `[FROM-CODE]`.)

4. **Separation of concerns: raw → normalized → validated → rendered** — Four
   strict waterlines:
   - **raw**: Parquet snapshot files + manifest, no transformations.
   - **normalized**: All `(servicer, remit_date)` Parquet files loaded into a
     structured frame with time anchors unified (`fctrdt`, etc.).
   - **validated**: Validators run SQL/pandas logic on the frame and emit
     `ValidatorResult` objects with per-cell highlight annotations.
   - **rendered**: `SheetPayload` → `XlsxRenderJob` carries all rendering
     parameters (fonts, colors, widths), fully decoupled from validator logic.
   (See ch 1.2 § 3 — end-to-end dataflow; ch 1.6 § 4–§ 7 — rendering
   attributes.)

5. **Time anchors derived once, centrally** — `remit_date`, `pre_date`,
   `fctrdt`, and `fctrdt_1m` are all derived once inside `MrcRemitFrame`
   construction. Downstream models consume the results; they do not re-derive
   or re-read from a `MrcDB` instance. (See ch 1.1 § 3 — `MrcDB.__init__` 4
   time anchors `[FROM-CODE]`.)

6. **`[VERIFY]` markers flow down with lineage** — Any attribute carrying
   `[VERIFY]` in Stage 1 (ch 1.6 § 9 checklist V1–V12) continues to carry
   `[VERIFY]` on the corresponding Stage 2 model field until the physical
   baseline confirms it, at which point it is upgraded to `[CONFIRMED]`.

7. **Snapshot manifest is the single source of truth (code sync-point)** —
   `SnapshotManifest` fields mirror `tools/freeze_snapshot.py::MANIFEST_ENTRY_TEMPLATE`
   exactly. Any change to that code constant **must** propagate to this
   document § 2.3 with a corresponding `decisions.md` entry. (Stated in
   `tools/freeze_snapshot.py` docstring: "the single source of truth".)

8. **No inter-validator data dependencies** — `ValidatorContext` provides each
   validator with an independent frame reference; validators may execute in
   parallel. Output ordering still matches the V1→V5 registration sequence.
   (See ch 1.2 § 3 — "5 validators have no data dependencies between them".)

### 2.2 `RawTableSnapshot`

| Attribute | Value |
|---|---|
| **Layer** | raw |
| **Purpose** | Metadata wrapper for a single Parquet file identified by `(servicer, remit_date, logical_name)`. The leaf node of the pipeline — no upstream back-reference since it *is* the raw source. |
| **Stage 1 anchor** | ch 1.1 § 4.2 — 13-sheet → Redshift table map; ch 1.1 § 6 — tables directly consumed by each of the 5 validators |
| **Servicer field** | `servicer: ServicerId` |

**Schema**:

```python
from dataclasses import dataclass
from datetime import date
from pathlib import Path

@dataclass(frozen=True)
class RawTableSnapshot:
    servicer:      ServicerId
    remit_date:    date
    logical_name:  str              # e.g. "portmrcremitloanlevelrecap"
    source_schema: str              # e.g. "mrc", "port"
    source_table:  str              # e.g. "portmrcremitloanlevelrecap"
    parquet_path:  Path             # snapshots/<remit_date>/raw/<servicer.lower()>/<logical_name>.parquet
    row_count:     int | None       # from manifest; None if not yet exported [FROM-CODE]
    column_names:  tuple[str, ...]  # column order matches Parquet schema
    # Lineage: this object is the leaf node — no upstream back-reference
```

**Notes**: `frozen=True` enforces immutability; `logical_name` corresponds to
`SnapshotManifest.logical_name` and can be used to reverse-look up full metadata.
Of the 13 ingested `mrc.portmrcremit*` tables, only **5** are directly consumed
by validators during report generation (ch 1.1 § 6 `[FROM-CODE]`); the other 8
serve the `basic_data_monthly_loan_common` ETL (ch 1.1 § 8).

### 2.3 `SnapshotManifest`

| Attribute | Value |
|---|---|
| **Layer** | raw |
| **Purpose** | Complete metadata record for one parquet export entry: source details, filter parameters, hashes, and Redshift session info. |
| **Stage 1 anchor** | `tools/freeze_snapshot.py::MANIFEST_ENTRY_TEMPLATE` (single source of truth) |
| **Servicer field** | Implicit in `source.schema` and `logical_name` prefix; consuming layer resolves via `ServicerId` |

> ⚡ **Code sync-point (AGENTS convention)** — The schema below must remain
> field-for-field consistent with `tools/freeze_snapshot.py::MANIFEST_ENTRY_TEMPLATE`.
> Any change to that constant **must** update this section and append a
> `decisions.md` entry.

**Schema** (direct mirror of `MANIFEST_ENTRY_TEMPLATE`, `MANIFEST_VERSION = "1.0"`):

```python
@dataclass(frozen=True)
class SnapshotManifestSource:
    type:   str   # "redshift" [FROM-CODE]
    schema: str   # e.g. "mrc", "port"
    table:  str   # e.g. "portmrcremitloanlevelrecap"

@dataclass(frozen=True)
class RedshiftSession:
    user:     str | None
    cluster:  str | None
    query_id: str | None

@dataclass(frozen=True)
class SnapshotManifest:
    # ── Mirror of tools/freeze_snapshot.py::MANIFEST_ENTRY_TEMPLATE ──
    logical_name:           str                    # e.g. "mrc_mrc_db_..."
    source:                 SnapshotManifestSource
    export_sql_path:        str                    # relative: _export_queries/<name>.sql
    filter:                 dict[str, str]          # placeholder → resolved value
    exported_at:            str | None             # ISO-8601 datetime string
    exporter:               str | None             # username or service account
    format:                 str                    # "parquet" [FROM-CODE]
    path:                   str                    # relative: parquet/<name>.parquet
    row_count:              int | None
    column_count:           int | None
    columns:                tuple[str, ...]
    sha256_file:            str | None
    sha256_canonical_rows:  str | None             # over rows sorted by primary key
    redshift_session:       RedshiftSession
    # ── Mirror end ────────────────────────────────────────────────────
```

**Notes**: `sha256_canonical_rows` is a row-set hash computed over rows sorted
by primary key — independent of file format, used for row-set equality checks.
`sha256_file` is the SHA-256 of the parquet file itself (see `tools/freeze_snapshot.py`
docstring `[FROM-CODE]`).

### 2.4 `MrcRemitFrame`

| Attribute | Value |
|---|---|
| **Layer** | normalized |
| **Purpose** | Assembles **all 8** upstream DataFrames needed for a `(MRC, remit_date)` run and unifies the 4 derived time anchors. Serves as the shared input carrier for all 5 validators — each validator selects the tables it needs from this frame. |
| **Stage 1 anchor** | ch 1.1 § 3 — 4 time anchors; ch 1.2 § 3 fig 1.2.3 — 8 upstream tables; ch 1.1 § 6 — per-validator table consumption |
| **Servicer field** | `servicer: ServicerId` (`= ServicerId.MRC` for this type; other servicers implement independent frame classes, see § 3) |

**Schema**:

```python
from pandas import DataFrame

@dataclass(frozen=True)
class MrcRemitFrame:
    servicer:       ServicerId    # = ServicerId.MRC for this type [FROM-CODE]
    remit_date:     date          # input parameter, e.g. date(2026, 4, 30)

    # ── Time anchors (centrally derived, see ch 1.1 § 3 [FROM-CODE]) ──
    pre_date:       date          # remit_date - MonthEnd(1), e.g. 2026-03-31
    fctrdt:         date          # get_fctrdt(remit_date), e.g. 2026-05-01
    fctrdt_1m:      date          # get_fctrdt(pre_date), e.g. 2026-04-01

    # ── Upstream tables (8, filtered to servicer + respective time anchor) ──
    portmonth:      DataFrame     # port.portmonth, fctrdt filter — ch 1.1 § 6
    portfunding:    DataFrame     # port.portfunding, left-join only — ch 1.1 § 6
    daily_curr:     DataFrame     # port.basic_data_daily_loan_common @ remit_date — ch 1.2 § 5.1
    daily_prev:     DataFrame     # port.basic_data_daily_loan_common @ pre_date — ch 1.2 § 4.1
    monthly_common: DataFrame     # port.basic_data_monthly_loan_common @ fctrdt — ch 1.2 § 5.1
    loanlevelrecap: DataFrame     # mrc.portmrcremitloanlevelrecap @ fctrdt — ch 1.1 § 6
    thirdparty_adv: DataFrame     # mrc.portmrcremit3rdpartyadvances @ fctrdt — ch 1.1 § 6
    corp_adv:       DataFrame     # mrc.portmrcremitcorpadvances @ fctrdt — ch 1.1 § 6
    escrow_adv:     DataFrame     # mrc.portmrcremitescrowadvances @ fctrdt — ch 1.1 § 6

    # ── Lineage ────────────────────────────────────────────────────────
    source_snapshots: tuple[RawTableSnapshot, ...]   # back-refs to the 8 raw snapshots
```

**Notes** `[VERIFY]`:
- Upstream lineage of `port.basic_data_daily_loan_common` and
  `port.basic_data_monthly_loan_common` is not decomposed in ch 1.1/1.2
  (treated as black-box, ch 1.1 § 8). `[VERIFY]` whether these tables need
  independent `SnapshotManifest` entries (DM-7).
- V5 (`mrc_other_check`) runs the same advances queries twice (`fctrdt` and
  `fctrdt_1m`), ch 1.2 § 6.3. `[VERIFY]` whether the frame holds two DataFrame
  variants or the validator over-queries internally (DM-6).

### 2.5 `MrcIntermediateCTEs`

| Attribute | Value |
|---|---|
| **Layer** | intermediate |
| **Purpose** | Captures the CTE-layer DataFrames for a given validator after SQL join execution but before the Python `asofdate` stamp. Enables UI drill-down ("which join row produced this diff?") and `[FOUND-DURING-STAGE2]` debugging. |
| **Stage 1 anchor** | ch 1.2 § 4–§ 6 — per-validator CTE structure; ch 1.4 § 2.1 — CTE alias asymmetry warning |
| **Servicer field** | `servicer: ServicerId` |

**Schema**:

```python
@dataclass(frozen=True)
class MrcIntermediateCTEs:
    servicer:     ServicerId
    remit_date:   date
    validator_id: str            # e.g. "mrc_adv_validation", "mrc_general_check"

    # ── CTE-layer outputs (field names aligned with SQL aliases) ──────
    # ⚠ Alias asymmetry [FROM-CODE] ch 1.4 § 2.1:
    #   mrc_adv_validation:  p1=prior, p2=current
    #   mrc_general_check:   p=current, p2=prior
    #   — Always interpret by asofdate filter, never by CTE name alone.
    cte_r:      DataFrame           # port.portmonth (driver table, all validators)
    cte_p1:     DataFrame | None    # prior-month daily (mrc_adv_validation only, ch 1.2 § 4.1)
    cte_p2:     DataFrame | None    # V3: current-month daily; V2: prior-month daily (ch 1.4 § 2.1)
    cte_p:      DataFrame | None    # current-month daily (mrc_general_check only, ch 1.2 § 5.1)
    cte_mc:     DataFrame | None    # port.basic_data_monthly_loan_common (V2 only)
    cte_f:      DataFrame | None    # port.portfunding (V2/V3/V4)
    join_result: DataFrame          # post-join, pre-Python stamp wide table

    # ── Lineage ────────────────────────────────────────────────────────
    parent_frame: MrcRemitFrame
```

**Notes**: V1 (`mrc_summary_check`) and V5 (`mrc_other_check`) use inline
SQL strings — no formal CTE structure. For these validators `cte_p1/p2/p/mc/f`
are `None` and `join_result` holds the SQL result DataFrame directly
(ch 1.2 § 6.1, § 6.3).

| **Layer** | validated (G2b / G3 harness output) |

class CellDiff:
    sheet_name:              str
    coord:                   CellCoord
    column_name:             str
    baseline_value:          Any
    stage2_value:            Any
    diff_type:               str    # "value"|"format"|"missing_cell"|"extra_cell"

    is_highlighted_stage2:   bool

class BaselineComparison:
    servicer:             ServicerId

`"format"` = number_format or style mismatch; `"missing_cell"` = Stage 2
lacks a cell present in baseline; `"extra_cell"` = Stage 2 has a cell absent
from baseline. Approved exemptions must be explicitly skipped in the harness
logic; the baseline file is never modified to accommodate them.

## 3. Servicer Extension Hooks

### 3.1 Design Intent

Stage 2 core models (`ValidatorContext`, `ValidatorResult`, `SheetPayload`,
`XlsxRenderJob`, `BaselineComparison`) are **servicer-agnostic** — they carry
a `servicer: ServicerId` field but do not branch on it internally. MRC-specific
frame types (`MrcRemitFrame`, `MrcIntermediateCTEs`) are decoupled from core
models through **generic parameterization**, so adding a new servicer requires
**no changes** to any core type. (Motivation: ch 1.1 § 5.1 — 5 scattered
`flow == 'mrc'` dispatch branches Stage 2 must eliminate.)

### 3.2 Generic Frame Interface

```python
from typing import TypeVar, Generic, Protocol

class RemitFrame(Protocol):
    """All servicer normalized frames must satisfy this protocol."""
    servicer:         ServicerId
    remit_date:       date
    source_snapshots: tuple[RawTableSnapshot, ...]

FrameT = TypeVar("FrameT", bound=RemitFrame)

@dataclass(frozen=True)
class ValidatorContextG(Generic[FrameT]):
    """Generic ValidatorContext; FrameT constrained to RemitFrame."""
    servicer:      ServicerId
    remit_date:    date
    frame:         FrameT
    validator_ids: tuple[str, ...]
    config:        dict[str, Any]

# MRC concrete instantiation:
# MrcValidatorContext = ValidatorContextG[MrcRemitFrame]
```

### 3.3 New Servicer Onboarding Checklist

To add a new servicer (e.g. Arvest), **no core types are modified** — only:

| Step | Action | Location |
|---|---|---|
| S1 | Add enum value to `ServicerId` (e.g. `ARVEST = "ARVEST"`) | This document § 2.1 + code |
| S2 | Implement `ArvestRemitFrame` satisfying `RemitFrame` protocol | New servicer module |
| S3 | Implement `ArvestIntermediateCTEs` if CTE tracing is needed | New servicer module |
| S4 | Register `servicer × validator_id → impl` in B4 validator-registry | `docs/stage2/4.0-validator-registry.*.md` |
| S5 | Provide `SnapshotManifest` entries (upstream table list for new servicer) | `tools/freeze_snapshot.py` + § 2.3 |
| S6 | Verify `XlsxRenderJob` rendering params are reusable or create servicer-specific overrides | B5 / B6 |

> `[VERIFY]` Arvest's frame fields and time anchor derivation have not yet
> been reverse-engineered (PrefectFlow source is in the read-only sibling repo,
> see AGENTS.md § 7); step S2 is a placeholder design (DM-10).

## 4. Lineage Contract

### 4.1 Complete Lineage Chain

Every final XLSX cell can be traced back to a raw table row through the
following object chain:

```
RawTableSnapshot (×8 tables)
  └── MrcRemitFrame.source_snapshots
        └── MrcRemitFrame (8 DataFrames)
              └── MrcIntermediateCTEs.parent_frame
                    └── MrcIntermediateCTEs (join_result)
                          └── ValidatorResult.source_ctes
                                └── ValidatorResult (df + cell_annotations)
                                      └── SheetPayload.source_result
                                            └── SheetPayload (body_cells at CellCoord)
                                                  └── XlsxRenderJob.source_payloads
                                                        └── XLSX cell (row, col)
```

### 4.2 Row-Level Back-Reference

Each DataFrame inside `MrcRemitFrame` preserves original Parquet row ordering
(no re-indexing). Each row in `ValidatorResult.df` maps back via `row_index`
(0-based pandas row index) to `source_ctes.cte_r` (the `port.portmonth` driver
rows), which are further traceable to a specific `MrcRemitFrame.portmonth` row
via `loanid` (see ch 1.2 § 4.2 — "the driver row set is `port.portmonth where
servicer='MRC' and fctrdt=...`").

### 4.3 UI Drill-Down Implications

The Stage 2 UI drill-down feature (plan.md § 7.3 B5) requires:

1. User clicks an XLSX cell → `CellCoord` locates entry in `SheetPayload.body_cells`.
2. `SheetPayload.source_result` → `ValidatorResult`, find matching `CellAnnotation`.
3. `CellAnnotation.row_index` → corresponding row in `ValidatorResult.df`.
4. `ValidatorResult.source_ctes.cte_r`'s `loanid` → original row in
   `MrcRemitFrame.portmonth`.
5. `MrcRemitFrame.source_snapshots` → `RawTableSnapshot.parquet_path` →
   original Parquet file.
