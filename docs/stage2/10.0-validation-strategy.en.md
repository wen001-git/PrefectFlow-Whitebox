# 10.0 Validation Strategy (Stage 2)

---

## 1. Why this exists

The ideal Stage 2 validation approach (G2a/G2b in `plan.md §§ 4.2/4.4`) requires:

1. A **frozen Parquet snapshot** of all Redshift inputs (`baselines/mrc/2026-04-30/input_snapshots/`).
2. A **frozen baseline XLSX** produced by running unmodified legacy code against that snapshot.
3. A **cell-identity diff** between the new system output and that frozen baseline.

This is reproducible, historically debuggable, and fully agent-driven once the snapshot exists.

**The blocker**: Redshift credentials, VPN access, and HashiCorp Vault tokens are unavailable to
the agent and not currently available to the operator for a scheduled export window. G2a — the
operator-driven export step — is **environment-blocked**.

Rather than freeze the entire project waiting for a credential window, this document codifies a
**temporary live comparison strategy** that:

- Unblocks Stage 2 implementation today.
- Preserves the long-term G2a/G2b/G3 ideal as an explicit migration target.
- Defines a new hard gate (**G2b-LIVE**) that gates all implementation work.

**Tooling status**: All G2a tooling is complete and waiting (`tools/freeze_snapshot.py` v2.2,
A1–A6 runbooks, `.env.example`). The moment a credential window opens, G2a can be closed by
running the operator runbook (`tools/docs/g2a-operator-runbook.{en,zh}.md`).

## 2. Two realities (long-term ideal vs temporary strategy)

Both layers coexist in code and docs. Do not delete or deprecate the long-term ideal columns — they
are the **migration target**.

| Layer | Long-term ideal (§§ 4.2/4.4) | Temporary reality (§ 9, this section) |
|---|---|---|
| Reproducibility substrate | Frozen Parquet snapshot under `baselines/mrc/<date>/input_snapshots/` | Live Redshift query each run (not reproducible across days) |
| Truth oracle | `baselines/mrc/<date>/validation_report.xlsx` frozen by G2b | Legacy `flow/remit_validation` MRC output, regenerated on demand |
| Validation method | Cell-identity diff vs frozen XLSX | Cell-identity diff vs **fresh legacy XLSX from same run instant** |
| Replayability | High — anyone with repo + LFS can re-run | Low — requires live Redshift; only operator can rerun |
| Historical debugging | Strong — past data preserved in LFS | Weak — yesterday's data is gone if Redshift changed |
| Status | Long-term recommended capability | **Temporary execution strategy — in force now** |

## 5. Operator workflow for a comparison run

Once G2b-LIVE is closed (C5 passes), the standard comparison workflow is:

### Step 1 — Run legacy system

The operator (person with Redshift VPN + Vault credentials) runs:

```bash
python tools/run_legacy_mrc.py --remit-date 2026-04-30
```

This wraps the legacy `flow/remit_validation` MRC entrypoint (see operator runbook A5:
`tools/docs/g2a-operator-runbook.{en,zh}.md`). Output: `baselines/mrc/2026-04-30/legacy_run/validation_report.xlsx`
plus a run-metadata sidecar JSON.

### Step 2 — Run new system

The operator (or agent, post-impl) runs:

```bash
python tools/run_newsystem_mrc.py --remit-date 2026-04-30
```

Initially this is the C3 stub that generates a synthetic XLSX. After Stage 2 implementation lands,
it invokes the real engine. Output: `baselines/mrc/2026-04-30/new_run/validation_report.xlsx`.

### Step 3 — Run the comparison

```bash
python tools/compare_validation.py compare \
  --legacy baselines/mrc/2026-04-30/legacy_run/validation_report.xlsx \
  --new    baselines/mrc/2026-04-30/new_run/validation_report.xlsx \
  --report baselines/mrc/2026-04-30/comparison_report.html
```

This invokes the C4 orchestrator (will land when C4 is complete).

### Step 4 — Read the HTML report and iterate

Open `comparison_report.html` in a browser. The report categorizes diffs as:

- **Value diffs** — cell value or formula divergence
- **Style diffs** — number-format, font, fill, border, alignment differences
- **Structural diffs** — missing/extra rows, merged-cell changes, column width / freeze-pane changes
- **Sheet-level diffs** — sheet order, missing/extra sheets, dimension changes

Address major value and structural diffs before tagging an impl PR as ready for review.
Each impl PR that touches a validator must include comparison-run evidence attached to the PR.

### Automated mode (shortcut)

```bash
python tools/compare_validation.py compare \
  --remit-date 2026-04-30 --servicer mrc --auto \
  --report baselines/mrc/2026-04-30/comparison_report.html
```

`--auto` invokes Steps 1–3 in sequence (requires operator credentials in `.env`).

## 6. Caveats explicitly accepted

By adopting this temporary strategy, the project explicitly accepts:

1. **Non-reproducibility across days** — Redshift source data can change; two comparison runs on
   different days may legitimately differ even with no code changes.
2. **Weak historical debugging** — if a regression surfaces later, we cannot replay the exact
   failing input without G2a (frozen snapshot).
3. **Operator bottleneck** — every meaningful validation cycle requires operator hands on
   Redshift credentials. Agent cannot run comparison autonomously.
4. **Shared-bug blindspot** — anything both legacy and new systems get wrong against truth (e.g.,
   a shared upstream bug) will not surface; only *divergence* between the two surfaces.
5. **Cross-day non-determinism false positives** — any non-deterministic ordering in legacy code
   will appear as false-positive diffs; the harness must tolerate or explicitly flag this.

These caveats are documented here — not papered over — so that any future contributor understands
the epistemic limits of comparison evidence produced under this strategy.

## 7. What this strategy does NOT replace

- **Stage 1 reverse-engineering docs** (`docs/mrc/1.0-toc → 1.6-baseline.{zh,en}.md`) remain
  the authoritative behavioral specification. They are living docs (AGENTS § 6.11) and continue
  to be updated as Stage 2 surfaces new findings.
- **The long-term G2a/G2b/G3 ideal** (§§ 4.2/4.4 of `plan.md`) is the migration target, not
  a deprecated design. When credentials become available, the project should migrate back.
- **Per-validator unit tests** — unit tests for individual validator logic are still required for
  all Stage 2 implementation work. The comparison harness supplements, not replaces, unit tests.

## 8. When credentials become available — migration path

When a Redshift credential window opens:

1. **Close G2a**: run the operator runbook (`tools/docs/g2a-operator-runbook.{en,zh}.md`).
   This populates `baselines/mrc/2026-04-30/input_snapshots/` with frozen Parquet snapshots.
2. **Close G2b-OLD**: run `tools/legacy_adapter.py` against the frozen snapshot to produce
   `baselines/mrc/2026-04-30/validation_report.xlsx` (the frozen baseline XLSX).
3. **Upgrade G3**: re-run the V1–V12 `[VERIFY]` sweep against the physical frozen XLSX, upgrading
   each item to `[CONFIRMED]` in `docs/mrc/1.6-baseline.{zh,en}.md` (AGENTS § 6.11 procedure).
4. **Retire the temporary strategy**: update `plan.md § 9`, this doc, and `progress.md` to
   reflect that G2a/G2b/G3 are now closed. G2b-LIVE comparison runs continue as a supplementary
   check, not the primary gate.
5. **Update `decisions.md`**: append a "strategy pivot retired" entry.
