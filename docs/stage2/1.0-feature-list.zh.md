## F1 — 参数选择器（选择 remit_date / servicer / trust / report type）

**一句话摘要**：用户在生成报告前，通过下拉/日期选择控件指定 `remit_date`、`servicer`、`trust` 和 `report type` 四个维度。

**用户故事**：作为基金运营分析师，我希望通过 UI 选择器指定报告所需的四个参数，以便精确生成我需要的报表组合，而不产生其他 Servicer 或日期的干扰数据。

**验收标准**：

- `remit_date` 选择器仅列出本地输入快照中已存在的日期（由 G2a 基准决定）。`[CONFIRMED]` 依赖 `baselines/mrc/<remit_date>/input_snapshots/_manifest.json` 存在。
- `servicer` 下拉列表枚举注册表中的所有 Servicer；`⏳ pending-analysis` 的 Servicer 显示禁用状态，并附链接指向对应的 `_pending.md`。`[VERIFY]` 各 Servicer 显示名与注册表 `display_name` 字段对应关系待确认。
- `trust` 选择器的可选值范围 `[VERIFY]`——Stage 1 文档尚未明确列出 MRC trust 字段的枚举值（见 `docs/mrc/1.4-fields.zh.md`）。
- `report type` 默认且仅显示"MRC Validation Report"；未来其他报表类型由 B6 可扩展性规格预留。
- 仅当四个参数均已设置时，"生成报告"按钮（F2）才变为可点击状态。
- 参数重置按钮可将所有选项恢复至默认值。

**范围标签**：`cross-servicer`（选择器本身是跨 Servicer 的基础设施）

**Stage 1 锚点**：see ch 1.0 § 2（范围定义）· ch 1.1 § 1（`remit_date` 时间锚点）

**requires-G2-baseline**：`no`（选择器 UI 本身不依赖基准，但显示哪些日期可用依赖 G2a 产物）

**开放问题 / `[VERIFY]` 标记**：
- `trust` 字段含义及枚举值在 Stage 1 文档中未出现，需结合 `mrc_validation.py` 及业务侧确认。`[VERIFY]`
- `report type` 在当前 PrefectFlow 中只有一种类型；未来扩展时需明确命名约定。`[VERIFY]`

## F2 — 报告生成器（点击生成 Validation Report）

**一句话摘要**：触发引擎按顺序执行全部 5 个 MRC 验证器，并将输出渲染为与旧系统 XLSX 单元格等价的工作簿。

**用户故事**：作为运营分析师，我希望点击"生成报告"后系统自动运行 MRC 验证逻辑并输出 XLSX，以便获得与旧系统完全一致的结果，无需手动运行 Python 脚本。

**验收标准**：

- 5 个 MRC 验证器按以下顺序执行：`mrc_summary_check` → `mrc_check_general_info` → `mrc_check_adv_balance` → `mrc_service_fee_check` → `mrc_other_check`（see ch 1.2 § 2）。`[CONFIRMED]` 依据 `remit_validation.py:134-144`。
- 生成的 XLSX 必须通过 `stage2-mrc-cell-identity-harness` 单元格等价检查（与基准 XLSX 在工作表名称、顺序、列顺序、表头、每个单元格值、高亮颜色上完全一致）。`[CONFIRMED]` 依据 ch 1.3 § 3–4。
- 生成过程中显示进度指示器（至少区分"处理中 / 完成 / 失败"三种状态）。
- 若输入快照数据缺失或损坏，提示清晰的错误信息并给出补救建议（如重新运行 G2a 导出）。
- 成功完成后，F3–F8 所有功能变为可用。

**范围标签**：`MRC-only`（生成逻辑与 MRC 验证器强耦合）

**Stage 1 锚点**：see ch 1.2 § 2（验证器执行顺序） · ch 1.3 § 3（渲染管道）

**requires-G2-baseline**：`yes`（需要 G2b 产出的基准 XLSX 才能运行单元格等价检查）

**开放问题 / `[VERIFY]` 标记**：
- 生成时间上限是否有明确要求？`[VERIFY]`
- 生成是否需要支持异步/后台运行？`[VERIFY]`

## F3 — 工作表逻辑查看器（查看每个 sheet 的生成逻辑）

**一句话摘要**：为 5 张 MRC 工作表分别展示对应的验证函数、SQL 模板、数据来源、连接键、过滤条件和分组逻辑。

**用户故事**：作为业务分析师或开发者，我希望查看每张工作表的生成逻辑，以便理解为什么某列/行呈现特定数值，而无需自行阅读 Python 源代码。

**验收标准**：

- 5 张工作表（`MRC_Summary_check` / `MRC_General_Check` / `MRC_Advance_Check` / `MRC_ServiceFee_Check` / `MRC_Adv_Info`）各有独立的逻辑面板。`[CONFIRMED]` 依据 ch 1.3 § 4.4。
- 每个面板展示：验证器函数名称、使用的 SQL 模板（如 `mrc_adv_validation`）、数据来源表、连接键（join key）、过滤条件（filter）、分组逻辑（group by）、计算字段。`[CONFIRMED]` 依据 ch 1.2 § 3。
- 面板内容与 Stage 1 文档严格一致，不允许出现文档与实现不一致的情况。
- 高亮列（highlight column）标注说明（哪些列会被条件格式着色，阈值为 `0`）。`[CONFIRMED]` 依据 ch 1.3 § 4.3。
- 非技术用户可以通过面板理解业务目的，无需阅读代码。

**范围标签**：`MRC-only`

**Stage 1 锚点**：see ch 1.2 § 3（各验证器逻辑）· ch 1.3 § 4（渲染合约）· ch 1.4（字段定义）· ch 1.5（验证规则）

**requires-G2-baseline**：`no`

**开放问题 / `[VERIFY]` 标记**：
- 每张工作表的逻辑面板是以弹出层（modal）还是侧边栏（side panel）展示？UI 设计（B4）待定。`[VERIFY]`

- `[VERIFY]` 是否支持单元格级别（行级）的值追溯（而非仅列级），该决策留待 B2 SRS 确认。

**范围标签**：`MRC-only`

## F6 — 逐步检查器（查看每一步的输入数据、中间结果、输出结果）

**一句话摘要**：对于一次报告生成运行，在每个验证器调用节点提供可浏览的输入 DataFrame、输出 DataFrame 及行级过滤摘要。

**用户故事**：作为排查差异的开发者，我希望在管道的每个步骤查看实际的输入数据、中间 DataFrame 和最终输出，以便准确定位偏差发生在哪个节点。

**验收标准**：

- 每个验证器步骤（5 个：`mrc_summary_check`→…→`mrc_other_check`）暴露：输入 DataFrame 的行数和列摘要、输出 DataFrame 的行数和列摘要、在该步骤中被过滤掉的行数（如有）。`[CONFIRMED]` 依据 ch 1.2 § 2 验证器执行流程。
- 数据以可分页表格形式展示（大数据集不一次性加载）。
- 逐步检查器可从 F3 工作表逻辑面板跳转打开。
- `[VERIFY]` 是否支持将当前运行的中间结果与基准运行（G2b 产物）进行逐行比对，性能影响待评估。
- `[VERIFY]` 中间数据捕获方式：运行时日志记录（logging）还是事后推导？待 B2 SRS 和 B5 开发计划确认。

**范围标签**：`MRC-only`

**Stage 1 锚点**：see ch 1.2 § 2（验证器执行顺序与 DataFrame 传递）

**requires-G2-baseline**：`yes`（与基准比对功能依赖 G2b 产物；仅查看当前运行中间结果则不依赖）

**开放问题 / `[VERIFY]` 标记**：
- 中间数据的持久化存储策略（内存 / 磁盘 / 数据库）待 B3 数据模型确认。`[VERIFY]`

## F7 — 验证结果溯源器（查看 validation pass / fail / exception 的原因）

**一句话摘要**：对每行验证结果，展示该行是否通过验证（pass / fail）、触发高亮的列、阈值和实际值；标记 exception 行（如有）的异常原因。

**用户故事**：作为基金运营分析师，我希望在报告中看到某行被高亮时，能立即查看是哪个字段超过了差异阈值、当前值是多少，以便快速判断是数据问题还是系统问题。

**验收标准**：

- 被高亮的单元格附有注解：超差列名、阈值（MRC 所有工作表均为 `0`）、实际绝对值。`[CONFIRMED]` 依据 ch 1.3 § 4.3（`diff_cell_format`，规则为 `abs(value) > 0`）。
- 未高亮行显示"PASS — 所有差异列均在阈值内"。
- `MRC_Summary_check` 和 `MRC_Adv_Info` 无高亮列，整体显示"PASS — 本工作表无差异列"。`[CONFIRMED]` 依据 ch 1.3 § 4.4（两张表 highlight_columns 为空列表）。
- `[VERIFY]` MRC 中是否存在 exception 行（即非 pass/fail 的第三种状态）？Stage 1 ch 1.5 规则目录需在 B2 SRS 阶段核实。
- 注解内容与 Stage 1 ch 1.5 验证规则目录严格一致，不允许出现与文档相矛盾的说明。

**范围标签**：`MRC-only`

**Stage 1 锚点**：see ch 1.3 § 4.3（高亮样式级联）· ch 1.5（验证规则目录）

**requires-G2-baseline**：`no`（判断逻辑是确定性的，仅依赖阈值规则，不依赖基准比对）

**开放问题 / `[VERIFY]` 标记**：
- MRC 是否有 exception 行的定义？ch 1.5 中的规则是否包含第三种状态？`[VERIFY]`

## F8 — 报告导出（导出 validation_report.xlsx）

**一句话摘要**：将已生成的报告以 `.xlsx` 文件形式下载到用户本地，文件与旧系统输出单元格等价。

**用户故事**：作为运营分析师，我希望将生成的报告导出为 XLSX 文件，以便以利益相关方熟悉的格式分享报告，且无需他们区分"新系统"与"旧系统"。

**验收标准**：

- 下载按钮输出文件名为 `validation_report.xlsx`（或含 `remit_date` 后缀的变体，待 B4 UI 设计确认）。
- 导出文件通过 `stage2-mrc-cell-identity-harness` 单元格等价检查：工作表名称/顺序/列顺序/表头/每个单元格值/高亮颜色/数字格式与基准 XLSX 完全一致。`[CONFIRMED]` 依据 ch 1.3 § 3–4（渲染管道 + 样式合约）。
- 导出功能仅在 F2 报告生成成功后可用（失败运行不允许导出）。
- 导出文件大小合理，不含不必要的冗余元数据。`[VERIFY]` 旧系统文件大小基准待 G2b 阶段测量。
- 单次生成、单次导出；不支持批量导出（批量导出属于超出范围，见第 9 节）。

**范围标签**：`MRC-only`

**Stage 1 锚点**：see ch 1.3 § 3（渲染管道）· ch 1.3 § 4（样式合约，含 money format、highlight 颜色 `ffc7ce`/`bccde9`）· ch 1.6（基准 XLSX 行为）

**requires-G2-baseline**：`yes`（单元格等价验证依赖 G2b 产出的基准 XLSX）

**开放问题 / `[VERIFY]` 标记**：
- 文件名是否包含 `remit_date` 后缀（如 `validation_report_20260430.xlsx`）？`[VERIFY]`
- 旧系统 XLSX 文件大小基准需在 G2b 阶段记录，以便验证新系统不额外膨胀。`[VERIFY]`

## 超出范围

以下内容**不属于** Stage 2 的功能范围，避免误将其纳入设计：

| 项目 | 说明 |
|---|---|
| 批量重新验证 | 一次生成多个 `remit_date` 的报告属于调度集成范畴，不在本范围内 |
| 调度集成 / 定时触发 | 与现有 Prefect Flow 调度器集成不在 Stage 2 范围内 |
| 非 MRC Servicer 的完整实现 | Arvest / CC5 / Selene / SLS 等 Servicer 的引擎实现待未来分析后再接入；本阶段仅预留占位符 |
| Redshift 实时连接 | Stage 2 使用 G2a 冻结的本地快照，不直接连接 Redshift |
| 报告编辑 / 修改 | 系统只读查看和导出，不支持在 UI 中修改单元格值 |
| 原始 ETL 管道接管 | Stage 2 是新系统，不替换或修改原 `PrefectFlow` 源码 |
| 用户权限 / 多租户 | 本阶段不引入认证授权体系 |

## 术语表

| 术语 | 定义 |
|---|---|
| **单元格等价（cell-identical）** | 新系统 XLSX 与旧系统基准 XLSX 在工作表名称、工作表顺序、列顺序、表头、每个单元格值、高亮颜色和数字格式上完全一致，由 `stage2-mrc-cell-identity-harness` 自动验收。（see ch 1.3 § 3；ch 1.6） |
| **验证器溯源（validator trace）** | 对一次报告运行，能够追溯每个验证器函数的调用参数、输入 DataFrame 状态和输出 DataFrame 状态，以及每一行最终结论（pass / fail / exception）的依据。（F7 的技术底座） |
| **数据血缘（lineage）** | 从 Servicer 原始文件到 XLSX 单元格的完整因果链路，包括：入仓 ETL 步骤、Redshift 暂存表、统一表、验证器 SQL/Python、DataFrame 字段转换、渲染格式化。（F5 的核心概念）（see ch 1.1–1.2） |
