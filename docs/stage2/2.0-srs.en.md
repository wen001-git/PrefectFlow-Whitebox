### 2.2 F2 — Report Generator

**Feature summary**: Triggers the engine to execute all 5 MRC validators in sequence and render their outputs as a cell-identical XLSX workbook; F3–F8 features unlock after success.

**Stage 1 anchors**: ch 1.2 § 2 (validator execution order) · ch 1.3 § 3 (rendering pipeline) · ch 1.6 (baseline contract)

---

#### FR-F2.1 — Validator execution order

| Attribute | Content |
|---|---|
| **Description** | The engine executes the 5 MRC validators in the following fixed sequence: `mrc_summary_check` → `mrc_check_general_info` → `mrc_check_adv_balance` → `mrc_service_fee_check` → `mrc_other_check`. |
| **Inputs** | G2a-frozen local input snapshots (`baselines/mrc/<remit_date>/input_snapshots/`); F1 four parameters |
| **Outputs** | DataFrame result from each of the 5 validators |
| **Preconditions** | G2a is closed; all four F1 parameters are filled; engine can access the local snapshot |
| **Postconditions** | 5 DataFrame results are produced and passed to the rendering pipeline (FR-F2.2) |
| **Error handling** | If a validator throws an exception, halt remaining validators, log the exception, report "Step X failed" in the UI with an error summary |
| **B1 trace** | B1 F2 acceptance criterion 1 `[CONFIRMED]` (`remit_validation.py:134-144`) |

---

#### FR-F2.2 — Cell-identical XLSX output

| Attribute | Content |
|---|---|
| **Description** | The generated XLSX workbook must pass the `stage2-mrc-cell-identity-harness` check and match the baseline XLSX on: sheet names, sheet order, column order, header rows, every cell value, highlight colors (`ffc7ce` / `bccde9`), and number formats. |
| **Inputs** | 5 DataFrame results from FR-F2.1 |
| **Outputs** | `validation_report.xlsx` workbook (in memory); cell-identity check result (pass/fail) |
| **Preconditions** | G2b is closed (baseline XLSX `baselines/mrc/<remit_date>/validation_report.xlsx` exists); G3 is closed (V1–V12 confirmed) |
| **Postconditions** | XLSX workbook passes cell-identity check; F8 export is enabled |
| **Error handling** | If cell-identity check fails, log a diff summary (sheet / row / column / expected / actual) and display "Render validation failed — view diff report" in the UI |
| **B1 trace** | B1 F2 acceptance criterion 2 `[CONFIRMED]` (ch 1.3 § 3–4); ch 1.6 baseline contract fully covered |

---

#### FR-F2.3 — Progress indicator

| Attribute | Content |
|---|---|
| **Description** | During report generation, the UI displays a live progress indicator distinguishing at minimum three states: `in progress` (per-validator execution phase) / `complete` / `failed`. |
| **Inputs** | Engine execution status event stream |
| **Outputs** | UI progress component state updates |
| **Preconditions** | F2 generation task has been started |
| **Postconditions** | State transitions to `complete` on success; to `failed` with an error entry point on failure |
| **Error handling** | If the engine process silently times out (`[VERIFY]` timeout threshold TBD), the UI prompts the user to cancel or continue waiting after the threshold is reached |
| **B1 trace** | B1 F2 acceptance criterion 3 |

---

#### FR-F2.4 — Input snapshot error handling

| Attribute | Content |
|---|---|
| **Description** | If loading the input snapshot reveals missing data or `_manifest.json` validation failure, the system surfaces a clear error message before starting any execution and suggests a remedy. |
| **Inputs** | Snapshot directory structure; `_manifest.json` content; sha256 verification results |
| **Outputs** | Error message (including name of the missing dataset); link / guidance for re-running the G2a export |
| **Preconditions** | F2 trigger conditions are met (FR-F1.5) |
| **Postconditions** | Validator execution is not started in the presence of data errors; error is logged |
| **Error handling** | Distinguish three error sub-types: (a) snapshot file absent; (b) file exists but sha256 mismatch; (c) row/column count inconsistency with manifest — each yields a distinct message and recommendation |
| **B1 trace** | B1 F2 acceptance criterion 4 |

---

#### FR-F2.5 — Downstream feature unlock

| Attribute | Content |
|---|---|
| **Description** | Only after F2 report generation **completes successfully** do the UI entry points for F3–F8 become enabled. |
| **Inputs** | F2 execution result state (success / failure) |
| **Outputs** | F3–F8 UI components transition from disabled to enabled |
| **Preconditions** | F2 execution has completed |
| **Postconditions** | Success: F3–F8 enabled. Failure: F3–F8 remain disabled; no empty-state content rendered |
| **Error handling** | After failure, the user may re-trigger F2; F3–F8 remain disabled with the tooltip "Generate a report successfully first" |
| **B1 trace** | B1 F2 acceptance criterion 5 |
