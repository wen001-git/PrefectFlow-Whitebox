## F1 — Parameter Picker (Select remit_date / servicer / trust / report type)

**One-line summary**: Before generating a report, the user specifies all four report dimensions via drop-down / date-picker controls.

**User story**: As a fund operations analyst, I want to select the four report parameters from a UI picker so that I generate the exact report combination I need without contaminating results from other servicers or dates.

**Acceptance criteria**:

- The `remit_date` picker lists only dates for which a local input snapshot already exists (determined by the G2a baseline artifacts under `baselines/mrc/<remit_date>/input_snapshots/_manifest.json`). `[CONFIRMED]` — snapshot manifest contract established in Stage 1 ch 1.6.
- The `servicer` drop-down enumerates all servicers from the registry; `⏳ pending-analysis` servicers appear disabled with a tooltip linking to their `_pending.md`. `[VERIFY]` — mapping between registry `display_name` and UI label TBD.
- The `trust` picker's valid values `[VERIFY]` — the MRC trust field enumeration is not explicitly catalogued in Stage 1 documents (see `docs/mrc/1.4-fields.en.md`); must be confirmed with business stakeholders.
- `report type` defaults to and currently shows only "MRC Validation Report"; future report types are reserved in B6 extensibility spec.
- The "Generate Report" button (F2) is enabled only when all four parameters are set.
- A "Reset" button returns all parameters to their default state.

**Scope tag**: `cross-servicer` (the picker itself is infrastructure shared across all servicers)

**Stage 1 anchors**: see ch 1.0 § 2 (scope definition) · ch 1.1 § 1 (`remit_date` time anchors)

**requires-G2-baseline**: `no` (the picker UI does not depend on the baseline; which dates are available depends on the G2a artifacts)

**Open questions / `[VERIFY]` markers**:
- The meaning and enumeration of the `trust` field do not appear in Stage 1 documentation; confirmation needed from `mrc_validation.py` and business stakeholders. `[VERIFY]`
- There is currently only one report type in PrefectFlow; naming convention for future types TBD. `[VERIFY]`

## F2 — Report Generator (Click to Generate Validation Report)

**One-line summary**: Triggers the engine to execute all 5 MRC validators in sequence and renders their outputs as a cell-identical XLSX workbook.

**User story**: As an operations analyst, I want to click "Generate Report" and have the system automatically run the MRC validation logic and produce the XLSX, so that I receive the same result the legacy system would have produced without running Python scripts manually.

**Acceptance criteria**:

- All 5 MRC validators are executed in the following order: `mrc_summary_check` → `mrc_check_general_info` → `mrc_check_adv_balance` → `mrc_service_fee_check` → `mrc_other_check` (see ch 1.2 § 2). `[CONFIRMED]` — per `remit_validation.py:134-144`.
- The generated XLSX passes the `stage2-mrc-cell-identity-harness` cell-identity check (sheet names, sheet order, column order, header rows, every cell value, highlight colors match the baseline). `[CONFIRMED]` — per ch 1.3 § 3–4.
- A progress indicator is shown during generation (at minimum: distinguishing "in progress / complete / failed" states).
- If input-snapshot data is missing or corrupted, a clear error message is shown with a suggested remedy (e.g., re-run the G2a export).
- All features F3–F8 become available after a successful generation run.

**Scope tag**: `MRC-only` (generation logic is tightly coupled to MRC validators)

**Stage 1 anchors**: see ch 1.2 § 2 (validator execution order) · ch 1.3 § 3 (rendering pipeline)

**requires-G2-baseline**: `yes` (cell-identity check requires the G2b baseline XLSX)

**Open questions / `[VERIFY]` markers**:
- Is there a specified time limit for report generation? `[VERIFY]`
- Does generation need to support async / background execution? `[VERIFY]`

## F3 — Sheet Logic Viewer (View the generation logic for each sheet)

**One-line summary**: For each of the 5 MRC sheets, displays the associated validator function, SQL template, data-source tables, join keys, filter conditions, and group-by logic.

**User story**: As a business analyst or developer, I want to view the generation logic for each sheet so that I understand why a particular column or row contains a given value without having to read the Python source code directly.

**Acceptance criteria**:

- Each of the 5 sheets (`MRC_Summary_check` / `MRC_General_Check` / `MRC_Advance_Check` / `MRC_ServiceFee_Check` / `MRC_Adv_Info`) has a dedicated logic panel. `[CONFIRMED]` — per ch 1.3 § 4.4.
- Each panel shows: validator function name, SQL template used (e.g., `mrc_adv_validation`), data-source tables, join key, filter conditions, group-by logic, computed fields. `[CONFIRMED]` — per ch 1.2 § 3.
- Panel content is strictly consistent with Stage 1 documentation; inconsistencies between doc and implementation are treated as P0 bugs.
- Highlight columns are annotated (which columns use conditional formatting, threshold = `0`). `[CONFIRMED]` — per ch 1.3 § 4.3.
- Non-technical users can understand the business purpose from the panel without reading code.

**Scope tag**: `MRC-only`

**Stage 1 anchors**: see ch 1.2 § 3 (per-validator logic) · ch 1.3 § 4 (rendering contract) · ch 1.4 (field definitions) · ch 1.5 (validation rules)

**requires-G2-baseline**: `no`

**Open questions / `[VERIFY]` markers**:
- Whether the sheet logic panel is rendered as a modal dialog or a side panel is deferred to B4 UI design. `[VERIFY]`

## F4 — Field Lineage Viewer (View the calculation logic for each field)

**One-line summary**: Click any column header to reveal the source table, SQL expression, transformation logic, and business meaning of that field.

**User story**: As an analyst auditing the report, I want to click on any column and see exactly how that value was calculated — source table, SQL expression, transformation — so that I can trace the root cause of a discrepancy without consulting the Stage 1 field catalog manually.

**Acceptance criteria**:

- Clicking any column header opens a field-detail pop-up showing: source table / source field, SQL expression or Python formula, transformation logic (including rounding rules such as `round_to=2`), and business meaning. `[CONFIRMED]` — per ch 1.4 field catalog.
- Derived columns (e.g., `servicefee_diff` on the ServiceFee sheet) show their calculation formula step by step. `[VERIFY]` — whether MRC has equivalent derived columns distinct from raw SQL outputs; to be confirmed during B2 SRS.
- All columns across all 5 sheets are covered (100% coverage).
- Field-detail content is consistent with Stage 1 ch 1.4; contradictions are P0 bugs.
- `[VERIFY]` Cell-level drill-down (row-level value tracing) vs column-level only — scope decision deferred to B2 SRS.

**Scope tag**: `MRC-only`

**Stage 1 anchors**: see ch 1.4 (full field definition catalog) · ch 1.5 (validation rules)

**requires-G2-baseline**: `no`

**Open questions / `[VERIFY]` markers**:
- Is cell-level (row-dimension) drill-down part of F4, or does it belong to F6 (step-through inspector)? `[VERIFY]`

## F5 — Data Lineage Viewer (View data flow lineage)

**One-line summary**: A visual diagram showing the end-to-end data lineage from raw servicer files to every XLSX cell.

**User story**: As a technical analyst, I want to view the full data lineage — from raw servicer data ingest through to the final XLSX cell — so that during troubleshooting I can pinpoint exactly where in the ETL chain a value is introduced or transformed.

**Acceptance criteria**:

- The lineage diagram covers the full chain: raw servicer data → ingest ETL → staging tables → unified Redshift tables → validator SQL / Python → DataFrame → XLSX cell. `[CONFIRMED]` — per ch 1.1 (raw data layer) and ch 1.2 (dataflow layer).
- Each node in the diagram is clickable and reveals the relevant SQL statement or Python code excerpt (with Stage 1 source-file line-range citations).
- The diagram supports filtering by sheet or by field to reduce visual noise.
- `⏳ pending-analysis` servicers appear as dashed "TBD (pending analysis)" placeholder nodes; they are not hidden. `[CONFIRMED]` — per AGENTS.md § 6.8 placeholder rule.
- Node-ID labels in the diagram follow AGENTS.md § 6.9 legend convention (T# / V# / SH# nodes must have a legend table). `[VERIFY]` — legend format to be confirmed in B4 UI design.

**Scope tag**: `cross-servicer` (the lineage viewer is shared infrastructure; MRC is the only currently populated branch)

**Stage 1 anchors**: see ch 1.1 § 1–3 (raw data sources) · ch 1.2 § 1–2 (end-to-end dataflow)

**requires-G2-baseline**: `no` (the static lineage graph does not depend on a live run)

**Open questions / `[VERIFY]` markers**:
- Visualization library (e.g., Mermaid, D3.js, custom SVG) TBD in B4 UI design. `[VERIFY]`
- The ingest ETL layer (`flow/servicer_data/`) is not fully covered in Stage 1 (which starts at Redshift). `[VERIFY]`

## F6 — Step-Through Inspector (View input data, intermediate results, and output results at each step)

**One-line summary**: For a generated report run, provides a browsable view of the input DataFrame, output DataFrame, and row-filtering summary at each validator invocation.

**User story**: As a developer debugging a discrepancy, I want to inspect the actual input data, intermediate DataFrames, and final output at each pipeline step so that I can identify exactly which node in the chain produced an unexpected result.

**Acceptance criteria**:

- For each of the 5 validator steps (`mrc_summary_check` → … → `mrc_other_check`), the inspector exposes: input DataFrame row count and column summary; output DataFrame row count and column summary; number of rows filtered at that step (if any). `[CONFIRMED]` — per ch 1.2 § 2 (validator execution flow and DataFrame handoffs).
- Data is rendered in a paginated browsable table (large datasets are not loaded all at once).
- The step inspector is accessible from the F3 sheet logic panel via a direct link.
- `[VERIFY]` Whether a row-level diff against a baseline run (G2b output) is supported — performance implications to be assessed.
- `[VERIFY]` Intermediate-data capture method (runtime logging vs post-hoc derivation) — to be confirmed in B2 SRS and B5 dev plan.

**Scope tag**: `MRC-only`

**Stage 1 anchors**: see ch 1.2 § 2 (validator execution sequence and DataFrame handoffs)

**requires-G2-baseline**: `yes` for the baseline-diff sub-feature; `no` for inspecting the current run's intermediate data only

**Open questions / `[VERIFY]` markers**:
- Persistence strategy for intermediate data (in-memory / disk / database) — to be decided in B3 data model. `[VERIFY]`

## F7 — Validation Trace (View reasons for validation pass / fail / exception)

**One-line summary**: For each row in the report, shows whether the row passed validation, which columns exceeded the diff threshold, the threshold value, and the actual value; marks exception rows with their reason where applicable.

**User story**: As a fund operations analyst, I want to understand why a specific row in the report is highlighted so that I can quickly determine whether it reflects a data issue or a system issue without consulting the Stage 1 rule catalog.

**Acceptance criteria**:

- Highlighted cells are annotated with: the column name that exceeded the threshold, the threshold value (always `0` for all MRC sheets), and the actual absolute value. `[CONFIRMED]` — per ch 1.3 § 4.3 (`diff_cell_format` rule: `abs(value) > 0`).
- Non-highlighted rows show "PASS — all diff-columns within threshold".
- `MRC_Summary_check` and `MRC_Adv_Info` show "PASS — no diff columns for this sheet" (both have empty `highlight_columns`). `[CONFIRMED]` — per ch 1.3 § 4.4 (both entries pass `highlight_columns=[]`).
- `[VERIFY]` Does MRC have exception rows (a third state distinct from pass/fail)? The Stage 1 ch 1.5 validation-rule catalog must confirm this during B2 SRS work.
- Annotation content is strictly consistent with Stage 1 ch 1.5; any contradiction between the UI annotation and the documented rule is a P0 bug.

**Scope tag**: `MRC-only`

**Stage 1 anchors**: see ch 1.3 § 4.3 (highlight style cascade) · ch 1.5 (validation rule catalog)

**requires-G2-baseline**: `no` (the pass/fail determination is deterministic from the threshold rules and does not require comparison against a baseline run)

**Open questions / `[VERIFY]` markers**:
- Does MRC define exception rows? Is there a third pass/fail/exception state in ch 1.5? `[VERIFY]`

## F8 — Report Export (Export validation_report.xlsx)

**One-line summary**: Downloads the generated report as an `.xlsx` file that is cell-identical to the legacy system's output.

**User story**: As an operations analyst, I want to export the generated report as an XLSX file so that I can share it with stakeholders in the format they expect, without them needing to distinguish between the "new system" and the "legacy system".

**Acceptance criteria**:

- The download button produces a file named `validation_report.xlsx` (or with a `remit_date` suffix; final convention to be confirmed in B4 UI design).
- The exported file passes `stage2-mrc-cell-identity-harness`: sheet names / sheet order / column order / header rows / every cell value / highlight colors / number formats match the baseline XLSX exactly. `[CONFIRMED]` — per ch 1.3 § 3–4 (rendering pipeline + style contracts).
- Export is available only after a successful F2 report generation; a failed run does not produce a downloadable file.
- The exported file size is reasonable — no unnecessary metadata inflation. `[VERIFY]` — baseline file size to be measured during G2b.
- One generation → one export; batch export is explicitly out of scope (see § Out of scope below).

**Scope tag**: `MRC-only`

**Stage 1 anchors**: see ch 1.3 § 3 (rendering pipeline) · ch 1.3 § 4 (style contracts — money format, highlight colors `ffc7ce` / `bccde9`) · ch 1.6 (baseline XLSX behavior)

**requires-G2-baseline**: `yes` (cell-identity verification requires the G2b baseline XLSX)

**Open questions / `[VERIFY]` markers**:
- Should the filename include a `remit_date` suffix (e.g., `validation_report_20260430.xlsx`)? `[VERIFY]`
- Legacy XLSX file size baseline needs to be measured during G2b so the new system can be verified not to bloat. `[VERIFY]`

## Out of Scope

The following are **explicitly excluded** from Stage 2 to avoid scope creep:

| Item | Rationale |
|---|---|
| Bulk re-validate (multiple `remit_date` values in one run) | Scheduling integration territory; out of scope. |
| Scheduled / automated triggering | Integration with the existing Prefect Flow scheduler is not in scope for Stage 2. |
| Full engine implementation for non-MRC servicers | Arvest / CC5 / Selene / SLS engine logic awaits future per-servicer analysis; Stage 2 reserves placeholder slots only. |
| Live Redshift connectivity | Stage 2 uses G2a frozen local snapshots; no direct Redshift connection in the new system. |
| Report editing / cell modification | The UI is read-only; users cannot modify cell values in the interface. |
| Takeover of the original ETL pipeline | Stage 2 is a new parallel system; the `PrefectFlow` source repo remains read-only and unmodified. |
| User authentication / multi-tenancy | No auth framework is introduced in Stage 2. |

## Glossary

| Term | Definition |
|---|---|
| **cell-identical** | The new system's XLSX matches the legacy baseline XLSX byte-for-byte on: sheet names, sheet order, column order, header rows, every cell value, highlight colors, and number formats — verified automatically by `stage2-mrc-cell-identity-harness`. (see ch 1.3 § 3; ch 1.6) |

| **lineage** | The complete causal chain from raw servicer files to the XLSX cell, including: ingest ETL steps, Redshift staging tables, unified tables, validator SQL / Python, DataFrame field transformations, and rendering formatting. (Core concept of F5) (see ch 1.1–1.2) |
| **G2a / G2b / G3** | Three hard gates that must all close before implementation begins. G2a = local input-snapshot freeze (user action); G2b = reproduce baseline XLSX from legacy code (agent, depends on G2a); G3 = V1–V12 assertions upgraded to `[CONFIRMED]` (agent, depends on G2b). | **pending-analysis servicer** | A servicer whose reverse-engineering analysis has not yet started (Arvest / CC5 / Selene / SLS / etc.). Retained as a visible placeholder in the registry, UI dropdown, and lineage diagram — never silently dropped. (AGENTS.md § 6.8) |
