## 1. 设计原则

1. **Servicer 判别字段（discriminator）无处不在** — 每个共享模型（从
   `RawTableSnapshot` 到 `XlsxRenderJob`）都携带 `servicer: ServicerId`
   字段。Stage 2 **禁止**在模型定义中硬编码字符串 `"MRC"`；servicer 身份通过
   枚举值注入，不在业务逻辑内分支。（见 ch 1.1 § 5.1——`flow == 'mrc'` 分
   派散落在 5 处，Stage 2 应消除此种散落。）

2. **阶段间帧不可变（lineage 保全）** — 管线的每个阶段（raw → normalized
   → intermediate → validated → payload → render-job）都产出一个**新的不可变
   对象**，而不是就地修改上一阶段的数据结构。每个对象持有对其直接上游的引用，
   使任何输出单元格的血缘都可以端到端追溯到原始表行。（见 ch 1.4 § 3.1——
   5 层血缘图：raw → CTE → projection → stamp → render。）

3. **显式单元格坐标（cell-coordinate carrying）** — `SheetPayload` 在构建时
   预计算每个单元格的 `(row, col)` 坐标，而不是在渲染期临时解析 DataFrame
   位置。这使 `BaselineComparison` 可以不依赖渲染器做 cell-level diff，也是
   UI 钻取功能的前置条件。（见 ch 1.6 § 3.1——每张 sheet 的列顺序与 tab 顺序
   由源码固定。）

4. **职责分离：raw → normalized → validated → rendered** — 四条水线严格隔离：
   - **raw**：Parquet 快照文件 + 清单，不做任何变换。
   - **normalized**：把同一 `(servicer, remit_date)` 的多张 Parquet 装入结构化
     帧，统一时间锚点（`fctrdt`、`pre_date` 等）。
   - **validated**：validator 在帧上执行 SQL/pandas 逻辑，产出带高亮标注的
     `ValidatorResult`。
   - **rendered**：`SheetPayload` → `XlsxRenderJob` 管理渲染参数（字体、颜色、
     列宽），与 validator 业务逻辑完全解耦。
   （见 ch 1.2 § 3——端到端数据流；ch 1.6 § 4–§ 7——渲染属性。）

5. **时间锚点集中推导** — `remit_date`、`pre_date`、`fctrdt`、`fctrdt_1m` 全部
   在 `MrcRemitFrame` 构造时一次性推导，下游模型只消费结果，不重复计算，也不
   再读 `MrcDB` 实例。（见 ch 1.1 § 3——`MrcDB.__init__` 的 4 个时间锚点。）

6. **`[VERIFY]` 标记随血缘向下传递** — Stage 1 中凡标注 `[VERIFY]` 的属性
   （ch 1.6 § 9 V1–V12 列表），在 Stage 2 对应的模型字段上继续携带 `[VERIFY]`
   标记，直到物理 baseline 验证后升级为 `[CONFIRMED]`。

7. **Snapshot 清单为单一真值来源（代码同步点）** — `SnapshotManifest` 的字段
   精确镜像 `tools/freeze_snapshot.py::MANIFEST_ENTRY_TEMPLATE`。任何对该代码
   常量的修改**必须**同步更新本文档 § 2.3，并在 `decisions.md` 留记录。（约定来
   自 `tools/freeze_snapshot.py` docstring——"Manifest entry schema is encoded
   in MANIFEST_ENTRY_TEMPLATE below and is the single source of truth."）

8. **Validators 之间无数据依赖** — `ValidatorContext` 为每个 validator 提供独立
   的帧引用；validators 可以并行执行，Stage 2 引擎无需按序排列（输出排序仍需
   保持 V1→V5 注册顺序）。（见 ch 1.2 § 3——"5 个 validator 之间无数据依赖"。）

|---|---|

**Schema**（Python 伪代码，attrs / pydantic 约定）：

```python
from enum import Enum

class ServicerId(str, Enum):
    MRC    = "MRC"      # 已在 Stage 1 逆向 [FROM-CODE]
```

### 2.2 `RawTableSnapshot`

| 属性 | 值 |
|---|---|
| **Layer** | raw |
| **Purpose** | 对一个 `(servicer, remit_date, logical_name)` 组合的单张 Parquet 文件的元数据包装。是管线中"最叶端"的对象——本身即原始数据源，不持有 back-reference。 |
| **Stage 1 anchor** | ch 1.1 § 4.2 — 13 张 sheet → Redshift 表映射；ch 1.1 § 6 — 5 个 validator 直接读取的表 |
| **Servicer field** | `servicer: ServicerId` |

**Schema**：

```python
from dataclasses import dataclass
from datetime import date
from pathlib import Path

@dataclass(frozen=True)
class RawTableSnapshot:
    servicer:      ServicerId
    remit_date:    date
    logical_name:  str         # e.g. "portmrcremitloanlevelrecap"
    source_schema: str         # e.g. "mrc", "port"
    source_table:  str         # e.g. "portmrcremitloanlevelrecap"
    parquet_path:  Path        # snapshots/<remit_date>/raw/<servicer.lower()>/<logical_name>.parquet
    row_count:     int | None  # from manifest; None if not yet exported [FROM-CODE]
    column_names:  tuple[str, ...]  # column order matches Parquet schema
    # Lineage: 本对象即叶节点，无上游 back-reference
```

**Notes**：`frozen=True` 强制不可变性；`logical_name` 对应 `SnapshotManifest.logical_name`，
可反向查找完整元数据。13 张 `mrc.portmrcremit*` 表中只有 **5 张**被 validator 直接消费
（ch 1.1 § 6 `[FROM-CODE]`）；其余 8 张在 `basic_data_monthly_loan_common` ETL 中使用（ch 1.1 § 8）。

### 2.3 `SnapshotManifest`

| 属性 | 值 |
|---|---|
| **Layer** | raw |
| **Purpose** | 一个 parquet 导出条目的完整元数据记录，包含来源、导出参数、哈希值及 Redshift 会话信息。 |
| **Stage 1 anchor** | `tools/freeze_snapshot.py::MANIFEST_ENTRY_TEMPLATE`（单一真值来源） |
| **Servicer field** | 隐含于 `source.schema` 与 `logical_name` 前缀；消费方通过 `ServicerId` 在外层解析 |

> ⚡ **代码同步点（AGENTS 约定）** — 以下 schema 必须与
> `tools/freeze_snapshot.py::MANIFEST_ENTRY_TEMPLATE` 字段一一对应。
> 任何对该常量的修改**必须**同步更新此处并在 `decisions.md` 留记录。

**Schema**（直接镜像 `MANIFEST_ENTRY_TEMPLATE`，`MANIFEST_VERSION = "1.0"`）：

```python
@dataclass(frozen=True)
class SnapshotManifestSource:
    type:   str   # "redshift" [FROM-CODE]
    schema: str   # e.g. "mrc", "port"
    table:  str   # e.g. "portmrcremitloanlevelrecap"

@dataclass(frozen=True)
class RedshiftSession:
    user:     str | None
    cluster:  str | None
    query_id: str | None

@dataclass(frozen=True)
class SnapshotManifest:
    # ── 镜像 tools/freeze_snapshot.py::MANIFEST_ENTRY_TEMPLATE ──────────
    logical_name:           str                   # e.g. "mrc_mrc_db_init_..."
    source:                 SnapshotManifestSource
    export_sql_path:        str                   # relative: _export_queries/<name>.sql
    filter:                 dict[str, str]         # placeholder → resolved value
    exported_at:            str | None            # ISO-8601 datetime string
    exporter:               str | None            # username or service account
    format:                 str                   # "parquet" [FROM-CODE]
    path:                   str                   # relative: parquet/<name>.parquet
    row_count:              int | None
    column_count:           int | None
    columns:                tuple[str, ...]
    sha256_file:            str | None
    sha256_canonical_rows:  str | None            # rows sorted by primary key
    redshift_session:       RedshiftSession
    # ── 镜像结束 ────────────────────────────────────────────────────────
```

**Notes**：`sha256_canonical_rows` 是排序后行级哈希，独立于文件格式，用于行集
级相等性验证；`sha256_file` 是 parquet 文件本体的 SHA-256（见 `tools/freeze_snapshot.py`
docstring `[FROM-CODE]`）。

### 2.4 `MrcRemitFrame`

| 属性 | 值 |
|---|---|
| **Layer** | normalized |
| **Purpose** | 把一个 `(MRC, remit_date)` 运行所需的**全部 8 张**上游 DataFrame 组合在一起，统一推导全部 4 个时间锚点。是 5 个 validators 的共同输入载体——每个 validator 从中取自己需要的表。 |
| **Stage 1 anchor** | ch 1.1 § 3 — 4 个时间锚点；ch 1.2 § 3 图 1.2.3 — 8 张上游表；ch 1.1 § 6 — validator 表消费关系 |
| **Servicer field** | `servicer: ServicerId`（此帧类型专为 MRC 设计；非 MRC servicer 实现独立帧，见 § 3） |

**Schema**：

```python
from pandas import DataFrame

@dataclass(frozen=True)
class MrcRemitFrame:
    servicer:        ServicerId   # = ServicerId.MRC for this type [FROM-CODE]
    remit_date:      date         # 输入参数，e.g. date(2026, 4, 30)

    # ── 时间锚点（集中推导，见 ch 1.1 § 3 [FROM-CODE]） ──────────────
    pre_date:        date         # remit_date - MonthEnd(1), e.g. 2026-03-31
    fctrdt:          date         # get_fctrdt(remit_date), e.g. 2026-05-01
    fctrdt_1m:       date         # get_fctrdt(pre_date), e.g. 2026-04-01

    # ── 上游表（8 张，全部过滤至 servicer + 对应时间锚点） ────────────
    portmonth:       DataFrame    # port.portmonth, fctrdt — ch 1.1 § 6
    portfunding:     DataFrame    # port.portfunding, left-join only — ch 1.1 § 6
    daily_curr:      DataFrame    # port.basic_data_daily_loan_common @ remit_date — ch 1.2 § 5.1
    daily_prev:      DataFrame    # port.basic_data_daily_loan_common @ pre_date — ch 1.2 § 4.1
    monthly_common:  DataFrame    # port.basic_data_monthly_loan_common @ fctrdt — ch 1.2 § 5.1
    loanlevelrecap:  DataFrame    # mrc.portmrcremitloanlevelrecap @ fctrdt — ch 1.1 § 6
    thirdparty_adv:  DataFrame    # mrc.portmrcremit3rdpartyadvances @ fctrdt — ch 1.1 § 6
    corp_adv:        DataFrame    # mrc.portmrcremitcorpadvances @ fctrdt — ch 1.1 § 6
    escrow_adv:      DataFrame    # mrc.portmrcremitescrowadvances @ fctrdt — ch 1.1 § 6

    # ── Lineage ──────────────────────────────────────────────────────────
    source_snapshots: tuple[RawTableSnapshot, ...]   # 8 个原始快照的 back-reference
```

**Notes** `[VERIFY]`：
- `port.basic_data_daily_loan_common` 与 `port.basic_data_monthly_loan_common`
  的上游血缘未在 ch 1.1/1.2 中拆解（视为黑盒，见 ch 1.1 § 8）。`[VERIFY]`
  这两张表是否需要在 `SnapshotManifest` 中单独列出（DM-7）。
- V5（`mrc_other_check`）对同一组 advances 表运行两次 SQL（`fctrdt` 和
  `fctrdt_1m`），见 ch 1.2 § 6.3。`[VERIFY]` 帧是否需要持有双份 DataFrame，
  还是由 validator 内部按 `fctrdt_1m` 过滤（DM-6）。

### 2.5 `MrcIntermediateCTEs`

| 属性 | 值 |
|---|---|
| **Layer** | intermediate |
| **Purpose** | 捕获某个 validator 执行 SQL join 后、Python stamp 之前的中间 DataFrame——即 CTE 层输出的显式快照。用于 UI 钻取（"这一行 diff 由哪个 join 产出？"）与调试。 |
| **Stage 1 anchor** | ch 1.2 § 4–§ 6 — 各 validator 的 CTE 结构；ch 1.4 § 2.1 — CTE 别名命名不对称警告 |
| **Servicer field** | `servicer: ServicerId` |

**Schema**：

```python
@dataclass(frozen=True)
class MrcIntermediateCTEs:
    servicer:     ServicerId
    remit_date:   date
    validator_id: str            # e.g. "mrc_adv_validation", "mrc_general_check"

    # ── CTE 层输出（字段名与 SQL 别名对齐） ──────────────────────────
    # ⚠ 别名不对称 [FROM-CODE] ch 1.4 § 2.1：
    #   mrc_adv_validation:  p1=上月, p2=当月
    #   mrc_general_check:   p=当月,  p2=上月
    #   — 始终以 asofdate 过滤为准，不要以 CTE 名为准
    cte_r:      DataFrame           # port.portmonth（所有 validators 共用驱动表）
    cte_p1:     DataFrame | None    # 上月 daily（mrc_adv_validation 专用，ch 1.2 § 4.1）
    cte_p2:     DataFrame | None    # V3: 当月 daily；V2: 上月 daily（见 ch 1.4 § 2.1）
    cte_p:      DataFrame | None    # 当月 daily（mrc_general_check 专用，ch 1.2 § 5.1）
    cte_mc:     DataFrame | None    # port.basic_data_monthly_loan_common（V2 专用）
    cte_f:      DataFrame | None    # port.portfunding（V2/V3/V4 用）
    join_result: DataFrame          # join 之后、Python stamp 之前的宽表

    # ── Lineage ──────────────────────────────────────────────────────────
    parent_frame: MrcRemitFrame
```

**Notes**：V1（`mrc_summary_check`）和 V5（`mrc_other_check`）用内联拼接 SQL，
无正式 CTE 结构；`cte_p1/p2/p/mc/f` 对这两个 validators 为 `None`，`join_result`
直接装填 SQL 结果 DataFrame（ch 1.2 § 6.1、§ 6.3）。

### 2.6 `ValidatorContext`

| 属性 | 值 |
|---|---|

### 2.8 `SheetPayload`

| 属性 | 值 |

| **Layer** | payload |
| **Purpose** | 一张 XLSX sheet 渲染所需的全部数据：表头、数据行及按 `(row, col)` 预计算的单元格坐标。渲染器以此为输入，无需再解析 DataFrame 位置。 |

| **Servicer field** | `servicer: ServicerId` |

**Schema**：

```python
@dataclass(frozen=True)
class CellCoord:
    row: int   # 1-based（行 1 = 表头）
    col: int   # 1-based

CellGrid = dict[CellCoord, Any]   # coord → cell value

@dataclass(frozen=True)
class SheetPayload:
    servicer:          ServicerId
    remit_date:        date
    sheet_name:        str             # e.g. "MRC_General_Check" — ch 1.6 § 3.1 [FROM-CODE]
    tab_order:         int             # 1-based；tab 顺序见 ch 1.6 § 3.1 [FROM-CODE]

    highlight_columns: frozenset[str]         # diff 列名集合 — ch 1.6 § 6.3 [FROM-CODE]
```

**Notes** `[VERIFY]`：`body_cells` 中 money 列的 `None`/`NaN` 强转为 `0` 见

`[VERIFY]`（DM-2、DM-3）。

```python
    border_style:     str   # "thin"

    # ── Freeze panes [VERIFY] ch 1.6 § 3.3 (DM-1) ───────────────────
```

## 3. Servicer 扩展钩子

### 3.1 设计意图

Stage 2 的核心模型（`ValidatorContext`、`ValidatorResult`、`SheetPayload`、
`XlsxRenderJob`、`BaselineComparison`）是 **servicer-agnostic** 的——它们持有
`servicer: ServicerId` 字段但不对其内容分支。MRC 专用帧类型（`MrcRemitFrame`、
`MrcIntermediateCTEs`）通过**泛型参数化**与核心模型解耦，使新 servicer 接入时
**无需修改**任何核心类型。（设计动机见 ch 1.1 § 5.1——5 处 `flow == 'mrc'` 散落
分支，Stage 2 应消除。）

### 3.2 泛型帧接口

```python
from typing import TypeVar, Generic, Protocol

class RemitFrame(Protocol):
    """所有 servicer 的 normalized 帧须满足此协议。"""
    servicer:         ServicerId
    remit_date:       date
    source_snapshots: tuple[RawTableSnapshot, ...]

FrameT = TypeVar("FrameT", bound=RemitFrame)

@dataclass(frozen=True)
class ValidatorContextG(Generic[FrameT]):
    """泛型 ValidatorContext；FrameT 约束为 RemitFrame。"""
    servicer:      ServicerId
    remit_date:    date
    frame:         FrameT
    validator_ids: tuple[str, ...]
    config:        dict[str, Any]

# MRC 具体化：
# MrcValidatorContext = ValidatorContextG[MrcRemitFrame]
```

### 3.3 新 servicer 接入清单

新增 servicer（例如 Arvest）时，**无需修改**核心类型，只需：

| 步骤 | 操作 | 位置 |
|---|---|---|
| S1 | 在 `ServicerId` 新增枚举值（e.g. `ARVEST = "ARVEST"`） | 本文档 § 2.1 + 代码 |
| S2 | 实现 `ArvestRemitFrame` 满足 `RemitFrame` 协议 | 新 servicer 模块 |
| S3 | 实现 `ArvestIntermediateCTEs`（若需要 CTE 追踪） | 新 servicer 模块 |
| S4 | 在 B4 validator-registry 注册 `servicer × validator_id → impl` | `docs/stage2/4.0-validator-registry.*.md` |
| S5 | 提供 `SnapshotManifest` 条目（新 servicer 的上游表列表） | `tools/freeze_snapshot.py` + § 2.3 |
| S6 | 验证 `XlsxRenderJob` 渲染参数可复用，或创建 servicer-specific overrides | B5 / B6 |

> `[VERIFY]` Arvest 的帧字段与时间锚点逻辑尚未逆向（PrefectFlow 源代码在只读
> 兄弟仓库中，见 AGENTS.md § 7）；S2 为占位设计（DM-10）。

## 4. 血缘契约

### 4.1 完整血缘链

每个最终 XLSX 单元格都可通过以下对象链追溯到原始表行：

```
RawTableSnapshot (×8 张表)
  └── MrcRemitFrame.source_snapshots
        └── MrcRemitFrame (8张 DataFrame)
              └── MrcIntermediateCTEs.parent_frame
                    └── MrcIntermediateCTEs (join_result)
                          └── ValidatorResult.source_ctes
                                └── ValidatorResult (df + cell_annotations)
                                      └── SheetPayload.source_result
                                            └── SheetPayload (body_cells at CellCoord)
                                                  └── XlsxRenderJob.source_payloads
                                                        └── XLSX 单元格 (row, col)
```

### 4.2 行级 back-reference

`MrcRemitFrame` 中的每张 DataFrame 保留 Parquet 文件的原始行顺序（不做重索引）。
`ValidatorResult.df` 中的每一行通过 `row_index`（0-based pandas 行索引）可以反
向对应到 `source_ctes.cte_r`（`port.portmonth` 的驱动行集），进一步通过 `loanid`
定位到 `MrcRemitFrame.portmonth` 的具体行（见 ch 1.2 § 4.2——驱动行集为
`port.portmonth where servicer='MRC' and fctrdt=...`）。

### 4.3 UI 钻取含义

Stage 2 UI 钻取功能（plan.md § 7.3 B5）需要：

1. 用户点击 XLSX 某单元格 → `CellCoord` 定位到 `SheetPayload.body_cells`。
2. `SheetPayload.source_result` → `ValidatorResult`，找到对应 `CellAnnotation`。
3. `CellAnnotation.row_index` → `ValidatorResult.df` 的对应行。
4. `ValidatorResult.source_ctes.cte_r` 中的 `loanid` → `MrcRemitFrame.portmonth`
   的原始行。
5. `MrcRemitFrame.source_snapshots` → `RawTableSnapshot.parquet_path` → 原始
   Parquet 文件。
